SQL1 pt
FA17 Q1.1

Given Deck(suit text, val text, score integer) primary key (suit, val). Which query returns 1/2 the score as a floating point number? One correct answer.

  • A. SELECT FLOAT(score)/2 FROM Deck
  • B. SELECT score/2 AS FLOAT FROM Deck
  • C. SELECT CAST(score AS FLOAT)/2 FROM Deck
  • D. SELECT (FLOAT)score/2 FROM Deck
  • E. SELECT score:FLOAT/2 FROM Deck
  • F. None of the above

Answer: C

Postgres uses CAST(attr AS TYPE) syntax for type casting. score is an integer, so score/2 does integer division. Casting first ensures float division.

Why others are wrong

A: FLOAT(score) is not valid SQL — looks like a function call syntax from other languages.
B: AS FLOAT creates a column alias named "FLOAT", doesn't cast.
D: C-style casting (FLOAT) is not valid SQL.
E: Colon syntax is not standard SQL casting.

Cheat Sheet Reference

Side 1, SQL section: CAST(col AS type) is listed under SQL expressions. Also remember that integer/integer = integer in SQL — always cast first for float results.

SQL5 pts
FA17 Q1.2

Create a view Hand with all possible distinct draws of two cards from the deck (without replacement) where order does not matter. E.g., include either (heart, jack, 11, spade, four, 4) or (spade, four, 4, heart, jack, 11), but not both. May be zero, one, or more correct answers.

  • A. ...WHERE NOT (draw1.suit = draw2.suit AND draw1.val = draw2.val)
  • B. ...WHERE (draw1.suit > draw2.suit OR (draw1.suit = draw2.suit AND draw1.val > draw2.val))
  • C. ...WHERE (draw1.suit < draw2.suit OR (draw1.suit = draw2.suit AND draw1.val < draw2.val))
  • D. ...SELECT DISTINCT ... WHERE NOT (draw1.suit = draw2.suit AND draw1.val = draw2.val)
  • E. ...WHERE (draw1.suit < draw2.suit AND draw1.val < draw2.val)

Answer: B, C

Both use lexicographic ordering on (suit, val) to break symmetry. For any pair of distinct cards, exactly one of draw1 > draw2 or draw1 < draw2 is true.

Why others are wrong

A: Excludes self-pairs but still includes both orderings: (heart,jack)+(spade,four) AND (spade,four)+(heart,jack). Doubles everything.
D: Same as A with DISTINCT — but since each row is already distinct (different column arrangement), DISTINCT doesn't help.
E: Uses AND instead of OR — misses pairs where suit1 < suit2 but val1 ≥ val2 (e.g., club-king, diamond-ace).

Cheat Sheet Reference

Side 1, SQL section: Self-joins for combining a table with itself. Key concept: to get unordered pairs from a self-join, impose a strict ordering like t1.col < t2.col (or use lexicographic ordering for composite keys).

SQL4 pts
FA17 Q1.3

Total score of a hand = sum of two card scores. Using the Hand view, which query assigns total score to each tuple? May be zero, one, or more correct answers.

  • A. Subquery with UNION then SUM(score) with GROUP BY
  • B. SUM(COALESCE(score1, score2)) with GROUP BY
  • C. Self-join Hand with Hand matching suit1=suit2, val1=val2, then h1.score1 + h2.score2
  • D. SELECT ..., score1 + score2 AS score FROM Hand GROUP BY suit1, val1, suit2, val2, score1, score2

Answer: D

Since each hand is uniquely identified by (suit1, val1, suit2, val2), grouping by all columns (including scores) produces one group per hand. score1 + score2 is a valid non-aggregate expression since both are in GROUP BY. This is equivalent to just SELECT ..., score1 + score2 FROM Hand.

Why others are wrong

A: UNION (set semantics) deduplicates — if score1 = score2 for some hand, one row is lost and SUM is wrong.
B: COALESCE returns the first non-NULL value — it doesn't add the two scores together.
C: Self-join on Hand with h1.suit1 = h2.suit2 matches card1 of one hand with card2 of another — not the same hand.

Cheat Sheet Reference

Side 1, SQL section: GROUP BY rule — every non-aggregate column in SELECT must appear in GROUP BY. Also: UNION = set (deduplicates) vs UNION ALL = bag (keeps all). Watch for the UNION deduplication trap!

Disks/Files7 pts
FA17 Q2

An 8 MB heap file on disk with 64 KB pages (1 MB = 1000 KB). No header pages. Use variable names: let A = answer to part A, B = answer to part B, etc.

A. How many pages are in the file?

B. If 24 directory entries fit on one page, how many additional directory pages do we need?

C. At least one page has space. Worst case I/Os to insert a new record?

D. Worst case I/Os to update a record given its primary key? (Heap file, fixed-length records)

E. Now the file is sorted by primary key. Worst case I/Os to update a record?

Answers

A. 8000 KB / 64 KB = 125 pages

B. ⌈A/24⌉ = ⌈125/24⌉ = 6 directory pages

C. B + 3 I/Os — Scan B directory pages to find a page with space, read that data page (1 IO), write it back (1 IO), write updated directory page back (1 IO). Since directory page is already in memory from the scan, only 1 write IO for it.

D. A + 1 I/Os — Must scan entire heap file (A pages) to find the record (no index), then write back (1 IO). Page directory doesn't help search — it only tracks free space.

E. log2(A) + 2A I/Os — Binary search to find the record: log2(A). Worst case: updating the sort key forces shifting records across all A pages (read all + write all = 2A IOs).

Key Insight

The "reuse results" format is common on CS 186 exams. Always express answers using previous variable names.

Critical trap in part D: Page directories help with inserts (finding free space) but NOT with searches. To find a record by primary key in a heap file, you must scan ALL data pages.

Critical trap in part E: Updating the sort key in a sorted file is expensive — you might need to shift everything. This is why sorted files have expensive inserts/deletes.

Cheat Sheet Reference

Side 1, IO Cost Model table: Heap file search = B (worst), insert = 2 IOs (read last page + write). Sorted file search = log2(B), but insert/delete requires shifting. The directory pages add overhead — check the "Page directory" note.

B+ Trees5 pts
FA17 Q3.1–3.3

Q3.1 (1 pt): What is the maximum fanout F of an order d B+-tree?

Q3.2 (2 pts): Each leaf holds F data entries. Max number of data entries an order d B+-tree of height 5 can store? (Height 1 = root only, height 2 = root + leaves)

Q3.3 (2 pts): Each leaf holds F entries. Minimum I/Os to check if a data entry exists in a B+-tree with 109 records? Express as a function of F.

Answers

Q3.1: F = 2d + 1 — Max occupancy is 2d keys per node, which means 2d+1 pointers (fanout).

Q3.2: F5 — At height h, the root has max F children, each interior node has max F children, and at height 5 we have 4 levels of fan-out before reaching F leaves, each with F entries: F × F × F × F × F = F5.

Q3.3: 1 + logF(109/F) I/Os — With max packing: 109/F leaf nodes. The tree has height 1 + logF(109/F). Each level requires 1 IO to traverse.

Key Insight

These formula questions appear on almost every midterm. Memorize:
• Max fanout = 2d + 1
• Max keys per node = 2d
• Height h tree: max entries at leaves = Fh (when we define height with root at 1)
• Min IOs for lookup = tree height (one IO per level)

Cheat Sheet Reference

Side 2, B+ Trees section: The formulas box has "Fanout F = 2d+1" and "Max entries at height h" directly. Also check "IO cost" formulas for search = tree height.

B+ Trees3 pts
FA17 Q3.4–3.6

Given this order 2 B+ tree:

Root: [8 | 13 | 20 | 25]
Leaves: [2,7] [8,11,12] [13,16] [21,22] [25,28,31,32]

Q3.4 (1 pt): How many nodes split when you insert 27?

Q3.5 (1 pt): After inserting 27, how many nodes split when you then insert 26?

Q3.6 (1 pt): After inserting 26 and 27, you insert 34, 35, 36, ..., 100. What keys are in the leftmost leaf?

Answers

Q3.4: 2 nodes split

Q3.5: 0 nodes split

Q3.6: [2, 7]

Detailed Trace for Q3.4 (Insert 27)

1. Navigate: 27 > 8, 13, 20, 25 → follow rightmost pointer to leaf [25, 28, 31, 32]
2. Leaf [25, 28, 31, 32] is full (order 2 = max 4 keys). Insert 27 → overflow [25, 27, 28, 31, 32]
3. Split leaf: [25, 27] and [28, 31, 32]. Copy up key 28 to parent.
4. Parent (root) [8, 13, 20, 25] is also full (4 keys). Must insert 28 → overflow [8, 13, 20, 25, 28]
5. Split root: [8, 13] and [25, 28]. Push up key 20 to new root.
Result: 2 splits (leaf + root)

Q3.5 (Insert 26)

After the split, the leaf containing [25, 27] has only 2 entries (max 4). Insert 26 → [25, 26, 27]. Fits! 0 splits.

Q3.6 (Insert 34–100)

All values 34–100 are larger than everything in the tree. They only affect the right side. The leftmost leaf [2, 7] is never touched.

Cheat Sheet Reference

Side 2, B+ Trees: "Leaf split: COPY UP middle key" and "Inner split: PUSH UP middle key." Remember: copy up = key stays in leaf; push up = key moves out of inner node. For order d, max keys = 2d. When a node has 2d+1 keys after insertion, split is needed.

Buffer Mgmt4 pts
FA17 Q5.3–5.5

Empty buffer pool with 4 frames. Access pattern: S E E M A W E S O M E (NOT including POSSUM)

Clock policy rules: Do NOT increment clock hand on a hit (only during page replacement).

Q5.3 (2 pts): What 4 pages are in the buffer after all accesses? (Alphabetical order)

Q5.4 (1 pt): What are the reference bits for each page? (Alphabetical order of pages, e.g., 1110)

Q5.5 (1 pt): How many buffer pool hits?

Answers

Q5.3: E, M, O, S

Q5.4: 1100 (E=1, M=1, O=0, S=0)

Q5.5: 2 hits

Clock Trace Walkthrough

Clock policy: clock hand sweeps frames. If ref bit = 1, set to 0 and advance. If ref bit = 0, evict and place new page with ref bit = 1. On a hit, just set ref bit = 1 (don't move clock hand).

Key steps: Fill buffer with S,E,M,A (all ref=1). Then:
• W (miss): clock at frame 0 (S, ref=1) → set 0, advance. Frame 1 (E, ref=1) → set 0, advance. Frame 2 (M, ref=1) → set 0, advance. Frame 3 (A, ref=1) → set 0, advance. Back to frame 0 (S, ref=0) → evict S, place W. Clock now at frame 1.
• E (hit): E is in buffer, set ref=1. Clock stays.
• S (miss): clock at frame 1 (E, ref=1) → set 0, advance. Frame 2 (M, ref=0) → evict M, place S. Clock at frame 3.
Continue tracing through O, M, E...
Final: E, M, S, O = alphabetically E, M, O, S with ref bits 1,1,0,0.

Cheat Sheet Reference

Side 2, Buffer Management: "Clock: ref > 0? Decrement & skip. ref = 0? Evict." For this exam variant, clock hand only moves during replacement (not on hits). Always note what variant the problem specifies!

Buffer Mgmt3 pts
FA17 Q5.6–5.8

Q5.6: Page A is loaded, never evicted, pin count = 7 at the end. True or False: Page A was accessed exactly 7 times.

Q5.7: True or False: The buffer manager decides when to set the dirty bit of a page.

Q5.8: 101 frames, 1000 accesses (900 to page P, 100 to 100 unique other pages). Repeated many times. True or False: MRU would have fewer misses than LRU.

Answers

Q5.6: FALSE — Pin count is the number of current users, not total accesses. A page could be accessed 8 times (pin count goes to 8) and then one process unpins (pin count = 7).

Q5.7: FALSE — The file/index management code (the caller) tells the buffer manager when a page is dirty, not the buffer manager itself.

Q5.8: FALSE — 101 unique pages total (P + 100 others), and the buffer has 101 frames. Everything fits! After the first pass, both MRU and LRU have 0 misses. No replacement ever needed.

Key Insights

Pin count trap: Pin count ≠ access count. Pin count tracks concurrent users. A process pins, uses, unpins. Multiple pins and unpins change the count independently of total accesses.

Dirty bit trap: The buffer manager acts on the dirty bit (writes dirty pages to disk before eviction) but doesn't set it. The higher-level code (requestor) sets the dirty bit when modifying a page.

Buffer size trap: When the buffer can hold ALL unique pages, replacement policy doesn't matter. Both have the same number of misses (just the initial cold misses).

Cheat Sheet Reference

Side 2, Buffer Management: "Pin count = # current users (NOT total accesses)" and "Dirty bit set by requestor on unpin, not by buffer manager" are both listed in the Key True/False section. Also: "If buffer ≥ unique pages, all policies are equivalent."

SQL4 pts
SP18 Q1.1

Schema: Player(pid, name, age), Relationship(pid1, pid2, type) where type is 'Friend' or 'Enemy'.

Mark all correct queries that return the average age of players who have at least one enemy.

-- A
SELECT AVG(age) FROM Player P, Relationship R
WHERE P.pid = R.pid1 AND R.type = 'Enemy'

-- B
SELECT AVG(age) FROM Player P
WHERE P.pid IN (
  SELECT pid1 FROM Relationship WHERE type = 'Enemy'
)

-- C
SELECT AVG(P.age) FROM Player P, Relationship R
WHERE P.pid = R.pid1
HAVING COUNT(CASE WHEN R.type = 'Enemy' THEN 1 END) > 0

-- D
SELECT AVG(age) FROM Player
WHERE pid IN (
  SELECT pid1 FROM Relationship WHERE type = 'Enemy'
  UNION
  SELECT pid2 FROM Relationship WHERE type = 'Enemy'
)
  • A. Join + WHERE filter
  • B. Subquery with IN on pid1
  • C. Join + HAVING with CASE
  • D. Subquery with UNION on pid1 and pid2

Answer: B, D

Detailed Analysis

A is WRONG: If a player has multiple enemies, they appear multiple times in the join, duplicating their age in the AVG calculation. AVG(age) over duplicated rows gives wrong answer.

B is CORRECT: The subquery produces a set of pid1 values who have enemies. IN checks membership — each player appears once in the outer query. AVG is correct.

C is WRONG: Missing GROUP BY! HAVING without GROUP BY treats the entire result as one group. Also, the WHERE join creates duplicates. The aggregate in HAVING acts on all joined rows, not per-player.

D is CORRECT: UNION collects both pid1 and pid2 from enemy relationships (deduplicates). A player is correctly identified as having an enemy whether they're pid1 or pid2.

Cheat Sheet Reference

Side 1, SQL Pitfalls: "Join before aggregate can duplicate rows → wrong AVG/SUM." Also: "WHERE vs HAVING: WHERE filters rows before grouping; HAVING filters after." Use IN subquery to avoid the duplicate-join problem.

SQL4 pts
SP18 Q1.2

Same schema. Find the number of enemies for EACH player (including those with 0 enemies).

-- A
SELECT P.name, COUNT(R.pid1) FROM Player P
LEFT JOIN Relationship R ON P.pid = R.pid1
WHERE R.type = 'Enemy' OR R.type IS NULL
GROUP BY P.name

-- B
SELECT P.name, COUNT(*) FROM Player P
LEFT JOIN Relationship R ON P.pid = R.pid1
WHERE R.type = 'Enemy' OR R.type IS NULL
GROUP BY P.name

-- C
SELECT P.name, COUNT(R.pid1) FROM Player P
LEFT JOIN Relationship R ON P.pid = R.pid1 AND R.type = 'Enemy'
GROUP BY P.name

-- D
SELECT P.name, COUNT(R.pid1) FROM Player P, Relationship R
WHERE P.pid = R.pid1 AND R.type = 'Enemy'
GROUP BY P.name
  • A. LEFT JOIN + WHERE filter for Enemy or NULL
  • B. LEFT JOIN + WHERE + COUNT(*)
  • C. LEFT JOIN with condition in ON + COUNT(R.pid1)
  • D. Inner join (cross join with WHERE)

Answer: C

The LEFT JOIN Trap

C is correct: Putting R.type = 'Enemy' in the ON clause means: "join only with enemy relationships." Players with no enemies get NULL for all R columns. COUNT(R.pid1) correctly returns 0 for NULLs.

A looks correct but has a subtle issue: Moving the type filter to WHERE actually converts the LEFT JOIN into an inner join for players with non-enemy relationships, BUT the OR R.type IS NULL preserves players with NO relationships at all. However, players who have ONLY friend relationships get filtered out — their R.type = 'Friend' (not NULL, not Enemy). So A is wrong for players with friends but no enemies.

B is wrong: Same problem as A, plus COUNT(*) counts NULL rows too, so players with 0 enemies would show count = 1.

D is wrong: Inner join excludes players with 0 enemies entirely.

Cheat Sheet Reference

Side 1, SQL Joins: "LEFT JOIN + condition in ON vs WHERE" — this is a classic exam question. Filter in ON = filter before joining (preserves all left rows). Filter in WHERE = filter after joining (can eliminate left rows). Also: COUNT(col) skips NULLs; COUNT(*) counts everything.

Disks/Files6.5 pts
SP18 Q2

Q2.1 True/False — mark ALL true statements:

A. Disk API provides read/write on full pages only
B. Disk can directly send records to CPU registers
C. SSD has better random read performance than HDD
D. Heap files are better than sorted files for fast writes
E. Heap files support faster range search than sorted files

Q2.2 Schema: Student(name CHAR(3), age INTEGER, bio TEXT, gpa CHAR(5)). Page size = 4096 bytes, 6-byte page footer. 4 bytes = integer, CHAR(n) = n bytes, TEXT = variable (can be 0).

Q2.2.1: What is the minimum possible record size (bytes)? Include a 4-byte record header.

Q2.2.2: Using minimum record size, how many records fit on 5 pages? (2-byte slot directory entries)

Answers

Q2.1: A, D are TRUE (C is also arguably true but the official answer is A, D)

Q2.2.1: 4 (header) + 3 (CHAR(3)) + 4 (INTEGER) + 0 (TEXT min) + 5 (CHAR(5)) = 16 bytes

Q2.2.2: Usable space per page = 4096 − 6 (footer) = 4090 bytes. Each record needs 16 bytes + 2 bytes (slot entry) = 18 bytes effective. Records per page = ⌊4090 / 18⌋ = 227. On 5 pages: 227 × 5 but the last calculation is ⌊4090/18⌋ = 227 per page × 5 = ~680 total records on 5 pages (actual: ⌊(4090)/(16+2)⌋ = 227, but the official answer accounts differently: ⌊4090/(16+2)⌋ per page).

Record Size Calculation Method

1. Start with record header size (given in problem)
2. Add each fixed-length field: CHAR(n) = n bytes, INTEGER = 4 bytes
3. For variable-length fields (TEXT, VARCHAR): minimum size = 0 bytes
4. Don't forget: slot directory entries take space too!

Records per page formula: ⌊(page_size − footer) / (record_size + slot_entry_size)⌋

Cheat Sheet Reference

Side 1, Disks/Files section: "Record size = header + sum of field sizes" and "Slotted page: footer + slot directory at bottom, records from top." The formula for records per page is in the IO Cost Model section.

B+ Trees4 pts
SP18 Q3.1–3.2

Q3.1 True/False about B+ trees (mark all true):

A. The number of I/Os needed to search for a key grows as O(log(N))
B. The fanout of a B+ tree with order d is 2d
C. Only the correct answer: log growth for search
D. Every time a leaf node is split, the height of the tree increases

Q3.2 Which is a valid order-2 B+ tree?

A. A tree with an underfull leaf (1 key in a non-root leaf)
B. A tree with key 10 in both root and inner node
C. A tree with an underfull inner node
D. A properly structured tree with all invariants maintained

Answers

Q3.1: C only (log growth for search)

Q3.2: D (the properly structured tree)

Common B+ Tree Traps

Fanout is 2d+1, not 2d. Order d means max 2d keys per node → max 2d+1 pointers (children). This is a frequent exam trap.

Leaf splits don't always increase height. Height only increases when the ROOT splits. A leaf split just adds a new leaf and pushes a key up. Only if that cascades all the way to a root split does height increase.

Occupancy invariants: Non-root nodes must have between d and 2d keys. Root can have 1 to 2d keys. Check every node in the tree — one violation makes it invalid.

Key can appear in leaf AND inner node (from copy-up), but the SAME key should NOT appear in two inner nodes on the same root-to-leaf path unless it's a copy-up from a leaf.

Cheat Sheet Reference

Side 2, B+ Trees: "Fanout = 2d+1" and "Occupancy: root 1–2d, others d–2d keys." Also: "Height increases ONLY on root split" is in the key facts section.

SP18 Q3.3

Bulk load keys [3, 5, 7, 8, 11, 19, 20] into an order-1 B+ tree with fill factor 2/3.

Q3.3a: What is the height of the resulting tree?

Q3.3b: How many inner (non-leaf) nodes?

Q3.3c: How many leaf nodes?

Q3.3d: What keys are in the leftmost leaf? Rightmost leaf?

Answers

Height: 2 (root + inner nodes + leaves = 3 levels, which is height 3 if counting from 1, but the official answer counts it as height 2 with root at level 0)
Actually: 3 inner nodes (1 root + 2 inner), 4 leaf nodes.

Leftmost leaf: [3, 5]

Rightmost leaf: [20]

Bulk Loading Method

1. Sort the data: Already sorted [3, 5, 7, 8, 11, 19, 20]
2. Fill factor 2/3 with order 1: Max keys = 2d = 2. Fill factor 2/3 × 2 = 1.33, so fill each leaf with 2 keys (round up from 1.33... actually fill factor means we fill to 2/3 capacity: ⌈2 × 2/3⌉ = 2 entries per leaf).
3. Pack leaves left to right: [3,5] [7,8] [11,19] [20]
4. Build inner nodes bottom-up: Copy up first key of each non-first leaf: 7, 11, 20 go up.
5. Inner nodes: [7] [20] with 11 pushed up to root → Root: [11]

Root: [11]
Inner: [7]    [20]
Leaves: [3,5] [7,8] [11,19] [20]

Cheat Sheet Reference

Side 2, B+ Trees: "Bulk load: sort first, fill leaves to fill factor, build inner nodes bottom-up." The fill factor determines how full each leaf is. Copy up the first key of each new leaf (except the first leaf) to the parent.

B+ Trees5 pts
SP18 Q3 Short Answers

A. What is the maximum fanout of an order (x+1) B+ tree?

B. Max keys at height 2 for order 2 B+ tree? (Height 1 = root only)

C. B+ tree of height 3 using Alternative 2 (key, recordID pairs at leaves). Equality search cost in I/Os?

Answers

A: 2(x+1) + 1 = 2x + 3

B: Root has max 2d=4 keys → 5 children (leaves). Each leaf has max 2d=4 keys. Total: 5 × 4 × 5 = 100 keys (Root fanout × leaf capacity × root fanout... actually: at height 2, root can have 5 children, each child is a leaf with max 4 keys = 5 × 4 = 20 entries. But if height 2 = root + leaves, then 5 leaves × 4 entries = 20. If they mean F2 = 52 = 25... The official answer is 100 which assumes height 2 means root with 2d+1=5 children, each inner node with 5 children, each leaf with 2d=4 entries: impossible at height 2. Let me reconsider: 5 × 4 = 20 at height 2. The official says 100, likely meaning height-2 = root(5 ptrs) → 5 inner nodes(5 ptrs each) → 25 leaves(4 each) = 100.)

C: 5 I/Os — 3 IOs to traverse tree (root + 2 inner levels) + 1 IO to read leaf + 1 IO to fetch actual record from heap (Alt 2 stores pointers, not records).

Alternative 2 Adds an Extra IO

This is a critical concept! B+ tree alternatives:
• Alt 1: Leaf stores actual record → equality search = tree height IOs
• Alt 2: Leaf stores (key, recordID) → equality search = tree height + 1 IO (to fetch record from heap)
• Alt 3: Leaf stores (key, list of recordIDs) → similar to Alt 2 but compressed

Height 3 + Alt 2 = 3 (traversal) + 1 (fetch record) but the official says 5, meaning they count leaf read separately: 3 inner node reads + 1 leaf read + 1 heap page read = 5.

Cheat Sheet Reference

Side 2, B+ Trees: "Alt 1: records at leaves. Alt 2: (key, rid) at leaves → +1 IO for heap fetch. Alt 3: (key, rid list)." Fanout formula is right at the top: F = 2d+1.

Table: Products(pid SERIAL, name VARCHAR(20), brand CHAR(8), price FLOAT). SERIAL = auto-increment integer (4 bytes when present, 0 when NULL). VARCHAR(20) = 0–20 bytes. CHAR(8) = 8 bytes. FLOAT = 8 bytes. Record header = 8 bytes. Page size = 1024 bytes, 10 data pages.

Q2.1: Which are possible record sizes? (Select all that apply)
A. 10   B. 20   C. 26   D. 36   E. 28   F. 46

Q2.3: How many records fit per page packed (fixed-size = max record = 44 bytes)?

Q2.4: How many records fit per page unpacked (variable-size, with 2-byte bitmap per record)?

Q2.5: Pages needed for linked-list heap file (with header page)?

Q2.6: Pages needed for page-directory heap file (20 entries per directory page)?

Answers

Q2.1: B, E, F (possible sizes are 20 to 28 when pid=NULL name=empty, up to 48 when pid present name=20 chars). Valid range: header(8) + pid(0–4) + name(0–20) + brand(8) + price(8) = 24–48. So 20 is possible if... actually with 8-byte header: min = 8+0+0+8+8 = 24, max = 8+4+20+8+8 = 48. Possible: 26 (B), 28 (E), 36 (D), 46 (F). Official answer: B, E, F meaning sizes 20, 28, 46 — this depends on exact header interpretation.

Q2.3: 16 records — ⌊1024/64⌋ = 16 (packed with max size)

Q2.4: 15 records — with 2-byte bitmap overhead: need to account for it

Q2.5: 401 pages — 400 data pages (10 pages × ... actually given 10 data pages) + 1 header page = 11. Wait — the problem says 10 data pages: linked list = 10 data + 1 header = 11. But official says 401 with a larger dataset.

Q2.6: 404 pages if using larger dataset.

Record Size Calculation

For record size questions, compute the range of possible sizes:
• Fixed fields: exact size (CHAR, INTEGER, FLOAT)
• Variable fields: min = 0, max = declared max (VARCHAR(n) = 0 to n)
• Nullable fields: can be 0 bytes if NULL
• Add the record header

Packed vs Unpacked layout:
• Packed: fixed-size slots = max record size, simple offset arithmetic
• Unpacked: variable-size with slot directory, each slot needs pointer overhead

Linked-list vs Page-directory:
• Linked list: 1 header page with pointers to first full page and first free page
• Page directory: ⌈data_pages / entries_per_dir_page⌉ directory pages

Cheat Sheet Reference

Side 1, Disks/Files: "Record size = header + fields" with the type sizes listed. "Packed: ⌊page_size / max_record⌋", "Unpacked: need slot directory entries." Heap file variants: linked-list (1 header + data pages) vs page-directory (⌈data/entries_per_page⌉ directory pages + data pages).

FA18 Q3.1–3.4

Schema: Books(bid, title, author, genre, yr), Libraries(lid, name, city), Checkouts(bid, lid, patron, date).

Q3.1 (3 pts): Find titles of books that have been checked out. Mark all correct:

A. SELECT title FROM Books, Checkouts WHERE Books.bid = Checkouts.bid
B. SELECT title FROM Books WHERE bid IN (SELECT bid FROM Checkouts)
C. SELECT DISTINCT title FROM Books, Checkouts WHERE Books.bid = Checkouts.bid
D. SELECT DISTINCT title FROM Books WHERE bid IN (SELECT bid FROM Checkouts)

Q3.2 (4 pts): For each library, count number of books checked out (including 0). Mark all correct:

A. SELECT L.name, COUNT(C.bid) FROM Libraries L LEFT JOIN Checkouts C ON L.lid = C.lid GROUP BY L.name
B. SELECT L.name, COUNT(C.bid) FROM Checkouts C RIGHT JOIN Libraries L ON L.lid = C.lid GROUP BY L.name
C. SELECT L.name, COUNT(*) FROM Libraries L LEFT JOIN Checkouts C ON L.lid = C.lid GROUP BY L.name
D. SELECT L.name, COUNT(C.bid) FROM Libraries L, Checkouts C WHERE L.lid = C.lid GROUP BY L.name

Q3.3 (4 pts): Find pairs of books with the same title (different bid). Mark correct:

A. SELECT B1.bid, B2.bid FROM Books B1, Books B2 WHERE B1.title = B2.title
B. SELECT B1.bid, B2.bid FROM Books B1, Books B2 WHERE B1.title = B2.title AND B1.bid < B2.bid
C. SELECT bid FROM Books WHERE title IN (SELECT title FROM Books WHERE bid != Books.bid)
D. SELECT B1.bid, B2.bid FROM Books B1 JOIN Books B2 ON B1.title = B2.title WHERE B1.bid != B2.bid

Q3.4 (5 pts): Find the library name with the most checkouts. Mark correct:

A. SELECT L.name FROM Libraries L, Checkouts C WHERE L.lid = C.lid
   GROUP BY L.name ORDER BY COUNT(*) LIMIT 1
B. SELECT L.name FROM Libraries L, Checkouts C WHERE L.lid = C.lid
   GROUP BY L.name HAVING COUNT(*) >= ALL(
     SELECT COUNT(*) FROM Checkouts GROUP BY lid)
C. SELECT name FROM Libraries WHERE lid IN (
   SELECT lid FROM Checkouts GROUP BY lid ORDER BY COUNT(*) DESC LIMIT 1)
D. SELECT L.name FROM Libraries L, Checkouts C WHERE L.lid = C.lid
   GROUP BY L.name HAVING COUNT(*) = MAX(COUNT(*))

Answers

Q3.1: C — Need DISTINCT because join can produce duplicates (book checked out multiple times). B is wrong because different books can share a title (no DISTINCT).

Q3.2: A, B — Both correctly use outer join to include libraries with 0 checkouts. COUNT(C.bid) correctly returns 0 for NULLs. C is wrong: COUNT(*) counts NULL rows as 1. D is wrong: inner join excludes libraries with 0 checkouts.

Q3.3: B — Self-join with B1.bid < B2.bid gives each pair exactly once. A includes self-pairs (same book) and duplicates. D includes both orderings. C has a correlated subquery issue.

Q3.4: B, C — B uses HAVING ≥ ALL (correct), C uses subquery with ORDER BY + LIMIT (correct). A is wrong: missing DESC on ORDER BY (gets least, not most). D is wrong: can't nest aggregate inside aggregate (MAX(COUNT(*))).

Pattern Recognition

DISTINCT trap: Join + aggregate can duplicate rows. Ask yourself: "Can a book appear multiple times in the join?" If yes, you need DISTINCT.

LEFT JOIN + COUNT trap: COUNT(*) vs COUNT(col). Use COUNT(nullable_column) with outer joins to get 0 for non-matching rows.

Self-join pair ordering: B1.bid < B2.bid is the canonical way to get unique unordered pairs.

"Most" queries: Two reliable patterns:
• HAVING COUNT(*) ≥ ALL (subquery)
• ORDER BY COUNT(*) DESC LIMIT 1

Cheat Sheet Reference

Side 1, SQL section covers all these: "DISTINCT needed after joins that duplicate", "COUNT(col) skips NULLs, COUNT(*) counts all", "Self-join pairs: use < for unique unordered pairs", "Max pattern: HAVING ≥ ALL or ORDER BY DESC LIMIT 1."

FA18 Q4

Alt 2 B+ tree (leaves have (key, recordID) pairs) on a heap file. Tree height = 3 (root at level 1, leaves at level 3). Heap file has 30 data pages.

Q4.1 (2 pts): Worst case I/Os for equality search?

Q4.2 (2 pts): Best case I/Os for equality search?

Q4.3 (3 pts): I/Os to insert a new record (assume leaf has space, heap page has space)?

Q4.4 (3 pts): I/Os to delete a record by name (no index on name)?

Q4.5 (2 pts): Maximum inserts before height change?

Q4.6 (2 pts): Minimum inserts to force height change?

Answers

Q4.1: 4 I/Os — 3 IOs to traverse tree (root → inner → leaf) + 1 IO to fetch record from heap page.

Q4.2: 3 I/Os — Key not found! Traverse 3 levels but record doesn't exist, so no heap page fetch needed.

Q4.3: 6 I/Os — Traverse tree to find correct leaf (3 reads) + read heap page (1) + write heap page with new record (1) + write leaf with new (key, rid) entry (1) = 6.

Q4.4: 34 I/Os — No index on name! Must scan ALL 30 heap pages + read each page (30 IOs) + for each matching record, update B+ tree... but worst case just scanning: must read all 30 data pages to find the record, then 1 write to delete from that page, then traverse tree to update index (3 reads + 1 write). Simplified: 30 (scan heap) + 1 (write heap) + 3 (traverse tree) + 1 (write leaf) = 34 IOs. Wait — without an index on name, we scan ALL 30 heap pages to find the record (30 IO), write back (1 IO) = 31, then update the B+ tree index: traverse 3 levels + write leaf = 3+1. But we know the key from the record, so 30+1+3 = 34.

Q4.5: 13

Q4.6: 4

IO Counting Strategy

For Alt 2 B+ trees, ALWAYS count the extra heap page IO:
• Search: tree traversal IOs + 1 (heap page read)
• Insert: tree traversal + heap page read + heap page write + leaf write
• Delete: find record first, then update both heap and tree

Best case search = key not found: You still traverse the tree but never fetch from the heap. This is a common trick question!

No index on search column = full heap scan: If deleting by name but the B+ tree is on a different column (like pid), you can't use the tree to find the record. Must scan all heap pages.

Cheat Sheet Reference

Side 2, B+ Trees: "Alt 2: +1 IO for heap fetch", "No index on query column = full scan (B IOs)." Side 1, IO Cost Model: heap file scan = B pages. For insert/delete, always count both read and write IOs separately.

FA18 Q4.7–4.8

Bulk load 12 records with keys [1–12] into an order 1 B+ tree with fill factor 1/2.

Q4.7: How many total pages (leaf + inner + root)?

Q4.8: What key is at the root?

Answers

Q4.7: 8 pages

Q4.8: Key 5 at root

Bulk Loading with Fill Factor 1/2

Order 1 → max 2 keys per leaf. Fill factor 1/2 → fill each leaf with ⌈2 × 1/2⌉ = 1 key per leaf (except the last leaf might have more).

But wait — fill factor 1/2 means we fill to 50% capacity. With max 2 entries: each leaf gets 1 entry.
12 records ÷ 1 per leaf = 12 leaves? That seems too many. Let's reconsider: fill factor 1/2 usually means fill to about half: 1 entry per leaf.
Actually, 12 entries with 1 per leaf = 12 leaves. Inner nodes (order 1, max 2 keys, 3 pointers): need ⌈12/3⌉ = 4 inner nodes. Then root points to 4: needs ⌈4/3⌉ = 2, and one root above...

The official answer of 8 pages suggests a different fill factor interpretation. With fill factor 1/2 and order 1: fill each leaf with 1 key. 12 keys → would be many leaves. The problem likely means the 12 records fill to 6 leaves (2 keys each, since 50% of max capacity of 2 is 1... hmm).

The answer of 8 total pages with key 5 at root means: 6 leaves + 2 inner nodes or similar configuration. Trust the official answer: 8 total pages, key 5 at root.

Cheat Sheet Reference

Side 2, B+ Trees: "Bulk load: sort → fill leaves to fill factor → build inner nodes bottom-up." Count total pages = leaves + inner nodes + root. The fill factor determines leaves-per-key ratio.

Buffer Mgmt4 pts
FA18 Q5.1

True/False — mark all true:

A. Each frame in the buffer pool is the same size as a disk page
B. The reference bit in Clock policy can have a value greater than 1
C. The buffer manager always reads pages from disk
D. When MRU and LRU produce different results, MRU always has more hits
E. If a file has fewer pages than buffer frames, MRU and LRU have the same number of hits

Answer: A, E are TRUE

A: TRUE — Buffer frames must match disk page size to hold one page per frame.
B: FALSE — Standard clock policy uses a single bit (0 or 1). PostgreSQL's variant uses a counter up to 5, but the standard algorithm uses a bit.
C: FALSE — If the page is already in the buffer pool (cache hit), no disk read needed.
D: FALSE — Neither MRU nor LRU is universally better. MRU is better for sequential flooding; LRU is better for temporal locality.
E: TRUE — If all pages fit in the buffer, no eviction ever happens. Both policies produce identical results after the initial cold misses.

Cheat Sheet Reference

Side 2, Buffer Management Key T/F: "Frame size = page size", "If buffer ≥ unique pages, all policies equivalent", "Neither MRU nor LRU always wins."

Buffer Mgmt4 pts
FA18 Q5.5

3 buffer frames. Sequential scan of 5 pages (A B C D E), repeated 3 times: A B C D E A B C D E A B C D E

Which policy has more hits: MRU or LRU?

Answer: MRU has more hits

LRU gets 0 hits (sequential flooding — every access evicts the page we'll need next).
MRU gets hits because it keeps older pages and evicts the most recently used one.

Sequential Flooding Explained

LRU with sequential scan (5 pages, 3 frames):
After filling buffer with [A,B,C]: D arrives, evict A (LRU). E arrives, evict B (LRU). A arrives again, evict C (LRU). We always evict the exact page we'll need next! 0 hits after initial load.

MRU with sequential scan:
After filling [A,B,C]: D arrives, evict C (MRU). Buffer: [A,B,D]. E arrives, evict D (MRU). Buffer: [A,B,E]. A arrives — HIT! B arrives — HIT! C arrives, evict E (MRU). Buffer: [A,B,C]. And the cycle continues with some hits each round.

This is the classic sequential flooding problem — the #1 argument for when MRU beats LRU.

Cheat Sheet Reference

Side 2, Buffer Management: "Sequential flooding: LRU gets 0 hits on repeated sequential scan when file > buffer. MRU keeps first pages, gets hits on re-scan." This concept is highlighted in the alert box on the cheat sheet.
