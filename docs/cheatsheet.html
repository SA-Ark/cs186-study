1. SQL FUNDAMENTALS

Query Evaluation Order

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

This is LOGICAL order, not execution order. Memorize it.

SELECT Basics

  • SELECT col1, col2 FROM T WHERE cond
  • DISTINCT — removes duplicate rows
  • ORDER BY col [ASC|DESC] — default ASC
  • LIMIT n — only deterministic WITH ORDER BY
  • Expressions: SELECT price * 1.1 AS tax_price

Aggregates

COUNT(*) COUNT(col) SUM(col) AVG(col) MIN(col) MAX(col)

COUNT(*) counts all rows. COUNT(col) counts non-NULL values.

GROUP BY Rules

CRITICAL: Every non-aggregated column in SELECT MUST appear in GROUP BY.

Wrong: SELECT name, MAX(gpa) FROM S

Right: SELECT dept, MAX(gpa) FROM S GROUP BY dept

WHERE vs HAVING

WHERE: filters rows BEFORE grouping. Cannot use aggregates.

HAVING: filters groups AFTER aggregation. CAN use aggregates.

SELECT dept, AVG(gpa) FROM S WHERE gpa > 2.0 GROUP BY dept HAVING AVG(gpa) > 3.5

Joins

LEFT JOIN trap: If left row has no match, right cols are NULL. Use IS NULL to find non-matches.

Subqueries

  • IN: WHERE col IN (SELECT ...)
  • EXISTS: WHERE EXISTS (SELECT 1 FROM ... WHERE ...)
  • NOT EXISTS: true when subquery returns 0 rows
  • ALL: WHERE col > ALL (SELECT ...) — true if greater than EVERY value
  • ANY: WHERE col > ANY (SELECT ...) — true if greater than at least ONE

Correlated: inner query references outer query column → re-runs per outer row (slow)

Uncorrelated: inner query is independent → runs once

CTEs & Views

  • CTE: WITH name AS (SELECT ...) SELECT ... FROM name
  • View: CREATE VIEW name AS SELECT ... — virtual table, query re-runs each use

SQL Traps from Past Exams

  • Integer division: 5/2 = 2, use CAST(x AS FLOAT) or x * 1.0
  • COUNT(DISTINCT col) ≠ COUNT(col)
  • NULL comparisons: NULL = NULL is UNKNOWN, not TRUE. Use IS NULL
  • NOT IN with NULLs: if subquery returns ANY NULL, entire NOT IN is UNKNOWN
  • DISTINCT + ORDER BY: ORDER BY cols must be in SELECT when DISTINCT is used

Storage Hierarchy

CPU Cache → RAM → SSD → HDD (fastest to slowest, smallest to largest)

File Types

Page = unit of disk I/O (4KB-8KB). Always read/write full pages.

Heap file = unordered pages. Easy insert, slow search.

Sorted file = pages sorted by key. Binary search for lookup.

Heap File Implementations

  • Linked list: header page → free pages list + full pages list. Must scan free list to find space.
  • Page directory: header pages store (pageID, free space) entries. Can jump directly to page with space.

Page Layout: Slotted Page

Records grow from TOP. Slot directory grows from BOTTOM.

Slot directory = array of (offset, length) pairs for each record.

Supports variable-length records + compaction.

Deletion: set slot pointer to null, compact later.

Record Layout

TypeLayout
Fixed-lengthFind field i at: base + sum of sizes of fields 0..i-1
Variable-lengthHeader with array of field offsets (pointers)

Null bitmap: 1 bit per field. Bit set = field is NULL.

Record Size Calculations

Fixed types: INTEGER = 4B, FLOAT = 4B, CHAR(n) = n bytes

Variable types: VARCHAR(n) = 0 to n bytes

Min size = header + fixed fields + VARCHAR at 0 bytes

Max size = header + fixed fields + VARCHAR at max bytes

Records/page: floor((page_size - page_overhead) / record_size)

4. B+ TREES

Structure

Order d: each node has d to 2d keys (root: 1 to 2d)

Interior nodes: keys + child pointers (fanout = pointers per node)

Leaf nodes: keys + record pointers, linked as doubly-linked list

All leaves at same depth (balanced)

Key Formulas

Max keys/node = 2d
Min keys/node = d (root: 1)
Max fanout (interior) = 2d + 1
Min fanout (interior) = d + 1 (root: 2)
Max leaf entries = 2d
Height h tree, max entries = (2d+1)^(h-1) * 2d

Search

Start at root. At each node: find correct child pointer by comparing search key. Follow pointer down. Reach leaf, scan for key.

Insert Algorithm

  1. Search to find correct leaf
  2. Insert key into leaf
  3. If leaf overflows (> 2d entries):

  Leaf split: split into two leaves. COPY UP middle key to parent.

  Interior split: split node. PUSH UP middle key to parent.

CRITICAL: Leaf = COPY up (key stays in leaf). Interior = PUSH up (key removed from node).

Delete Algorithm

  1. Find key in leaf, remove it
  2. If leaf underfull (< d entries):

  Try redistribute from sibling (borrow entry)

  If can't redistribute → merge with sibling + pull down parent key

May cascade up. Interior key deletion: pull separator down during merge.

Bulk Loading

  1. Sort all records by key
  2. Fill leaf pages to fill factor (e.g., 2/3 full, 3/4 full)
  3. Build parent nodes bottom-up

Fill factor f: entries per leaf = floor(2d * f)

Number of leaf pages = ceil(total_records / entries_per_leaf)

Much faster than inserting one at a time!

Data Alternatives

AltLeaf StoresNotes
1Actual recordsIndex IS the data file. Must be clustered.

3(key, list of recordIDs)Compact for duplicates. Can be clustered or not.

Clustered vs Unclustered

Clustered: leaf order = physical record order on disk. Sequential reads for range queries.

Unclustered: leaf order ≠ physical order. Random IOs per record in range query!

Only ONE clustered index per table (data sorted one way only).

Height typically 2-3 for realistic databases. Root often cached in buffer → -1 IO.

Composite Index Prefix Rule

Index on (A, B, C):

✓ Can use for: WHERE A=x | WHERE A=x AND B=y | WHERE A=x AND B=y AND C=z

✗ Cannot use for: WHERE B=y | WHERE C=z | WHERE B=y AND C=z

Must constrain the leftmost prefix of the composite key.

5. SPATIAL & VECTOR INDEXES

Why Needed

B+ trees handle 1D ordered data. Multi-dimensional queries (2D points, geographic data, vector similarity) need specialized indexes.

KD Trees

Binary tree alternating split dimension at each level.

Level 0: split on X. Level 1: split on Y. Level 2: split on X. Etc.

Each node stores a data point and partitions space into non-overlapping regions.

Search: at each node, compare query's dimension-value with node's split value. Go left if less, right if greater.

Insert: search to find position, add node.

KD Tree Nearest Neighbor

  1. Traverse to find candidate nearest point
  2. Backtrack: at each ancestor, check if other subtree could contain closer point
  3. If distance to split plane < current best distance, search other subtree too

App: RAG in LLMs — documents as vectors, find nearest neighbors for relevant context.

R-Trees

Like B+ trees for multi-dimensional data.

Interior nodes contain MBRs (Minimum Bounding Rectangles)

Leaf nodes contain actual spatial objects + bounding boxes

Key difference from KD: R-tree rectangles CAN OVERLAP

Search: check which MBRs overlap with query region, recurse into all matching children.

Insert: choose subtree whose MBR needs LEAST EXPANSION (smallest area increase).

6. BUFFER MANAGEMENT

Buffer Pool Structure

Array of frames in RAM. Each frame holds one page.

Each frame tracks: Page ID, Dirty Bit, Pin Count

Page Operations

OpAction
Pin (request)If in buffer: pin++. If not: read from disk, pin=1. If buffer full: evict page (pin=0 only)
Unpin (release)pin--. Optionally set dirty=true if page was modified.
EvictionPin count MUST be 0. If dirty: write to disk first. Then replace.

Pin count > 0 means page CANNOT be evicted!

Dirty bit is set by the REQUESTOR on unpin, NOT by the buffer manager.

Replacement Policies

LRU (Least Recently Used)

  • Evict page whose last access is oldest
  • Problem: SEQUENTIAL FLOODING — scanning file > buffer size evicts useful pages

MRU (Most Recently Used)

  • Evict page most recently accessed
  • Better for sequential scans — evicts just-finished page, keeps useful cached pages

Clock Policy (Used by PostgreSQL)

Circular buffer + clock hand + reference counter per frame.

On access: set ref counter (Postgres: increment up to max 5)

On eviction:

  1. Look at frame under clock hand
  2. If ref > 0: decrement ref, advance hand (give second chance)
  3. If ref = 0: EVICT this page, load new page here

Approximates LRU but cheaper to maintain.

Clock Trace Example

3 frames, hand at position 0:

Frame: [A(ref=1), B(ref=1), C(ref=0)]

Request D (not in buffer, need to evict):

Hand@0(A): ref=1 → set ref=0, advance

Hand@1(B): ref=1 → set ref=0, advance

Hand@2(C): ref=0 → EVICT C, load D, hand stays at 2 (or advances to 0)

Sequential Flooding Detail

Scenario: buffer has 3 frames, scanning file with pages P1,P2,...,P10

LRU: each new page evicts oldest → buffer churns through all pages → 0 hits

MRU: after buffer full, each new page evicts most recent → P1,P2 stay cached → useful if accessed again by other queries

Key True/False for Exams

  • Pin count = number of current users, NOT total accesses → TRUE
  • Buffer manager sets dirty bit → FALSE (requestor does)
  • A page with pin count 0 and dirty bit can be evicted → TRUE (write to disk first)
  • MRU always better than LRU → FALSE (MRU better for sequential, LRU better general)
  • Clock with max ref=1 approximates LRU → TRUE
  • Sequential flooding happens with MRU → FALSE (happens with LRU)
