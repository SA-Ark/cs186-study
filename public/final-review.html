Section 1: SQL (8 Questions)

Core SQL semantics, aggregation, joins, subqueries, and set operations.

Question 1

Write the conceptual evaluation order of the following SQL statement:

SELECT dept, COUNT(*)
FROM Students
WHERE gpa > 3.0
GROUP BY dept
HAVING COUNT(*) > 5
ORDER BY dept
LIMIT 3;
Solution

The conceptual (logical) evaluation order is:

  1. FROM — identify the table(s)
  2. WHERE — filter individual rows
  3. GROUP BY — form groups
  4. HAVING — filter groups
  5. SELECT — compute output expressions
  6. ORDER BY — sort the result
  7. LIMIT — truncate to N rows

Key insight: This is logical order, not the order the clauses appear in the SQL text. The optimizer may execute differently, but semantically this is what happens.

Question 2

Given a table Boba(shop_name TEXT, drink TEXT, price FLOAT, rating INT), write a query to find shop names where the average rating across all their drinks is above 4.0, ordered by average rating descending.

Solution
SELECT shop_name, AVG(rating) AS avg_rating
FROM Boba
GROUP BY shop_name
HAVING AVG(rating) > 4.0
ORDER BY avg_rating DESC;

Key insight: The HAVING clause filters after grouping (operates on groups), while WHERE filters before grouping (operates on individual rows). You cannot use column aliases in HAVING in standard SQL.

Question 3

What is wrong with this query?

SELECT name, dept, MAX(gpa)
FROM Students
WHERE COUNT(*) > 5
GROUP BY dept;
Solution

Two errors:

  1. name is not in GROUP BY but appears in SELECT without an aggregate function. Every non-aggregated column in SELECT must appear in GROUP BY.
  2. COUNT(*) cannot appear in WHERE. Aggregate functions go in HAVING, not WHERE. WHERE operates on individual rows before groups are formed, so aggregates are undefined there.

Corrected version:

SELECT dept, MAX(gpa)
FROM Students
GROUP BY dept
HAVING COUNT(*) > 5;
Question 4

Given Students(sid, name, gpa) and Enrollments(sid, cid, grade), write SQL using NOT EXISTS to find students who have never enrolled in any course.

Solution
SELECT s.name
FROM Students s
WHERE NOT EXISTS (
    SELECT 1
    FROM Enrollments e
    WHERE e.sid = s.sid
);

Key insight: NOT EXISTS is a correlated subquery — the inner query references s.sid from the outer query. For each student, it checks whether any enrollment row exists. SELECT 1 is conventional (the actual value selected doesn't matter, only whether any row is returned).

Question 5

What does the following query return?

SELECT *
FROM A LEFT OUTER JOIN B ON A.id = B.aid
WHERE B.aid IS NULL;
Solution

All rows from A that have no matching row in B.

How it works:

  1. The LEFT OUTER JOIN preserves all rows from A. For rows in A with no match in B, the B columns are filled with NULL.
  2. The WHERE B.aid IS NULL filter then keeps only those unmatched A rows.

Pattern: This is the standard "anti-join" pattern — find rows in one table with no corresponding row in another. Equivalent to NOT EXISTS or NOT IN (but safer than NOT IN with NULLs).

Question 6

Why is NOT IN dangerous when the subquery might return NULL values? What should you use instead?

Solution

If the subquery returns any NULL value, the entire NOT IN condition evaluates to UNKNOWN (not FALSE) for every row. This means no rows pass the filter and the query returns empty results.

Why? SQL uses three-valued logic. x NOT IN (1, 2, NULL) becomes x != 1 AND x != 2 AND x != NULL. The last comparison is always UNKNOWN, and anything AND UNKNOWN is at best UNKNOWN, never TRUE.

Use NOT EXISTS instead — it handles NULLs correctly because it only checks whether any row is returned, not the actual values.

Question 7

What is the difference between UNION and UNION ALL? When would you prefer each?

Solution
  • UNION — removes duplicates (set semantics). Requires a deduplication step (sort or hash).
  • UNION ALL — keeps all rows including duplicates (bag semantics). No deduplication overhead.

Prefer UNION ALL when:

  • You know there are no duplicates between the two result sets, or
  • Duplicates are acceptable in your use case.

It is faster because it skips the expensive deduplication step.

Question 8

Write a query using a CTE (Common Table Expression) that finds the department with the highest average GPA, then returns all students in that department.

Schema: Students(sid, name, dept, gpa)

Solution
WITH TopDept AS (
    SELECT dept
    FROM Students
    GROUP BY dept
    ORDER BY AVG(gpa) DESC
    LIMIT 1
)
SELECT s.*
FROM Students s, TopDept t
WHERE s.dept = t.dept;

Key insight: A CTE (WITH ... AS) defines a temporary named result set. It makes complex queries more readable by breaking them into logical steps. The CTE is evaluated first, then referenced in the main query like a table.

Section 2: Disks, Files & Records (4 Questions)

Record layout, page capacity, heap files, and I/O cost analysis.

Question 9

Given the following table definition, compute the minimum and maximum record sizes.

CREATE TABLE Products(
    id          INTEGER PRIMARY KEY,
    name        VARCHAR(10) NOT NULL,
    price       INTEGER NOT NULL,
    description CHAR(20)
);

Assume: Header = 8 bytes, INTEGER = 4 bytes.

Solution

Minimum record size:

  • Header: 8 bytes
  • id (INTEGER): 4 bytes
  • name (VARCHAR(10)): 0 bytes (minimum for variable-length)
  • price (INTEGER): 4 bytes
  • description (CHAR(20)): 20 bytes (always fixed)

Min = 8 + 4 + 0 + 4 + 20 = 36 bytes

Maximum record size:

  • Header: 8 bytes
  • id (INTEGER): 4 bytes
  • name (VARCHAR(10)): 10 bytes (maximum for variable-length)
  • price (INTEGER): 4 bytes
  • description (CHAR(20)): 20 bytes (always fixed)

Max = 8 + 4 + 10 + 4 + 20 = 46 bytes

Key rule: CHAR(n) is ALWAYS n bytes (fixed-length, padded). VARCHAR(n) is 0 to n bytes (variable-length, stores only actual content).

Question 10

Pages are 1 KB (1024 bytes). Fixed-length records are 64 bytes each. The page uses a packed layout with a bitmap (1 bit per slot, rounded up to the nearest byte).

How many records fit per page?

Solution

Let n = number of records. We need:

64n + ceil(n/8) ≤ 1024
  • Try n = 15: 64(15) + ceil(15/8) = 960 + 2 = 962 ≤ 1024 -- fits
  • Try n = 16: 64(16) + ceil(16/8) = 1024 + 2 = 1026 > 1024 -- does not fit

Answer: 15 records per page.

Formula: For packed layout with bitmap, each record costs its size + 1/8 byte for the bitmap. Don't forget to round the bitmap size up to the nearest whole byte.

Question 11

You have 10,000 records and a heap file with 50 records per page capacity. All data pages are half full.

How many data pages are in the heap file?

Solution

Half full means each page holds: 50 / 2 = 25 records.

Total pages: 10,000 / 25 = 400 data pages.

Key insight: Fill factor directly impacts space usage and scan costs. A half-full file uses 2x the pages of a fully packed file. This is common after many insertions and deletions in a heap file.

Question 12

An 8 MB heap file has 64 KB pages. How many I/Os for:

  1. Worst-case equality search on a primary key (heap file)?
  2. Same file but sorted — equality search using binary search?
Solution

Number of pages: 8 MB / 64 KB = 8000 KB / 64 KB = 125 pages.

(a) Heap worst case: 125 I/Os

Must scan every page because records are unordered. The target could be on the last page (or not exist at all).

(b) Sorted binary search: ceil(log2(125)) = 7 I/Os

Binary search halves the search space each step. 27 = 128 > 125, so at most 7 page reads.

Takeaway: Sorting reduces equality search from O(N) to O(log N) pages. This is why indexes exist — they provide the benefits of sorting without requiring the entire file to be sorted.

Section 3: B+ Trees (5 Questions)

Tree structure, splits, bulk loading, composite indexes, and I/O analysis.

Question 13

In a B+ tree of order d = 2:

  1. What is the max number of keys per node?
  2. What is the minimum number of keys per non-root node?
  3. What is the max fanout (children) of an interior node?
Solution
  1. 2d = 4 keys maximum per node.
  2. d = 2 keys minimum per non-root node (occupancy invariant).
  3. 2d + 1 = 5 children maximum per interior node (one more pointer than keys).

B+ tree invariants: Each node has between d and 2d keys (except the root, which can have as few as 1). Interior nodes have one more pointer than keys. These invariants guarantee the tree stays balanced and at least 50% full.

Question 14

Explain the critical difference between leaf node splits and interior node splits during B+ tree insertion.

Solution

Leaf split: COPY UP

  • The middle key is copied up to the parent.
  • The key stays in the leaf because leaf keys are needed to find actual data records.
  • After the split, the copied key appears in both the leaf and the parent.

Interior split: PUSH UP

  • The middle key is pushed up to the parent.
  • The key is removed from the interior node — it only serves as a routing guide, not as data.
  • After the split, the pushed key appears only in the parent.

Mnemonic: Leaves hold data, so keys must stay (COPY). Interior nodes are just routers, so keys can move (PUSH).

Question 15

You have 1,000 records to bulk load into an Alt-2 B+ tree. The fill factor is 2/3 and each leaf can hold a maximum of 6 entries.

How many leaf pages result from the bulk load?

Solution

With fill factor 2/3, each leaf holds: 6 × 2/3 = 4 entries.

Leaf pages: 1000 / 4 = 250 leaf pages.

Key insight: Bulk loading sorts the data first, then fills leaves left-to-right to the specified fill factor. The fill factor < 1 leaves room for future insertions without immediate splits. Alt-2 means the leaf entries are (key, recordId) pairs — the data lives separately in heap file pages.

Question 16

A composite index exists on (last_name, first_name). Which of these queries can use the index?

  1. WHERE last_name = 'Smith'
  2. WHERE first_name = 'John'
  3. WHERE last_name = 'Smith' AND first_name = 'John'
  4. WHERE first_name = 'John' AND last_name = 'Smith'
Solution
  1. Yes — matches the leftmost prefix of the composite key.
  2. No — first_name is NOT the leftmost prefix. The index sorts by last_name first, so without constraining last_name, the index cannot narrow the search.
  3. Yes — matches the full composite key.
  4. Yes — same as (c). The SQL optimizer reorders predicates to match the index order. The order of AND conditions in the WHERE clause doesn't matter.

Prefix rule: A composite index on (A, B, C) can be used for queries constraining A, or A+B, or A+B+C — always from the leftmost column. Think of it like a phone book sorted by last name then first name.

Question 17

A height-3 Alt-2 B+ tree has a clustered index. A range query returns 100 records spread across 5 consecutive leaf pages and 5 consecutive data pages.

How many I/Os? What if the index were unclustered?

Solution

Clustered index:

  • 3 I/Os — traverse tree from root to first matching leaf
  • 5 I/Os — read 5 consecutive leaf pages (following sibling pointers)
  • 5 I/Os — read 5 consecutive data pages (clustered = data is sorted in same order as index)

Total: 3 + 5 + 5 = 13 I/Os

Unclustered index:

  • 3 I/Os — traverse tree
  • 5 I/Os — read leaf pages
  • Up to 100 I/Os — each record could be on a different data page

Total: 3 + 5 + up to 100 = up to 108 I/Os

Takeaway: Clustered indexes are dramatically better for range queries because consecutive index entries point to consecutive data pages. Unclustered indexes can degrade to nearly one I/O per record.

Section 4: Spatial Indexes (3 Questions)

KD trees, R-trees, and spatial data structures.

Question 18

In a KD tree, the root splits on X at value 5. Its left child splits on Y at value 3. You are searching for point (4, 6).

Describe the search path.

Solution
  1. Root (X = 5): Query x-coordinate is 4. Since 4 < 5, go LEFT.
  2. Left child (Y = 3): Query y-coordinate is 6. Since 6 > 3, go RIGHT.
  3. Continue searching in the right subtree (next split dimension would be X again).

Key rule: KD trees alternate splitting dimensions at each level. Level 0 = X, Level 1 = Y, Level 2 = X, Level 3 = Y, etc. At each node, compare the query's coordinate in the current dimension against the split value.

Question 19

How does an R-tree choose which subtree to insert a new point into?

Solution

Choose the subtree whose MBR (Minimum Bounding Rectangle) needs the least expansion (smallest increase in area) to accommodate the new point.

This heuristic minimizes overlap between sibling MBRs and reduces wasted space, leading to more efficient searches. If the point falls inside an existing MBR, that subtree requires zero expansion and is preferred.

Contrast with B+ trees: B+ trees use key comparison to route insertions (deterministic). R-trees use a heuristic (least area expansion) because spatial data doesn't have a single linear ordering.

Question 20

Name 3 key differences between KD trees and R-trees.

Solution
  1. Branching factor: KD trees are binary (each node has 2 children). R-trees are multi-way balanced trees (like B+ trees, with high fanout).
  2. Region overlap: KD tree regions are non-overlapping (space is cleanly partitioned). R-tree MBRs can overlap, which means a search may need to explore multiple subtrees.
  3. Balance guarantee: KD trees are not guaranteed balanced (depends on insertion order). R-trees are always balanced (all leaves at same depth, like B+ trees).

Additional differences:

  • R-trees are more disk-friendly (node = page, high fanout reduces tree height).
  • KD trees are better for in-memory nearest-neighbor search.
  • KD trees split on one dimension per level; R-trees use bounding rectangles covering all dimensions simultaneously.

Section 5: Buffer Management (5 Questions)

LRU, MRU, Clock policy, pin counts, dirty bits, and sequential flooding.

Question 21

MRU Policy. Given 4 buffer frames and the access pattern:

I  L  O  V  E  D  B  Y  I  P  P  E

What are the final pages in the buffer? How many cache hits?

Solution

MRU evicts the most recently used (accessed) page when the buffer is full.

Step  Access  Buffer State         Action
 1     I      [I, _, _, _]          miss — load I
 2     L      [I, L, _, _]         miss — load L
 3     O      [I, L, O, _]         miss — load O
 4     V      [I, L, O, V]         miss — load V (buffer full)
 5     E      [I, L, O, E]         miss — evict V (MRU), load E
 6     D      [I, L, O, D]         miss — evict E (MRU), load D
 7     B      [I, L, O, B]         miss — evict D (MRU), load B
 8     Y      [I, L, O, Y]         miss — evict B (MRU), load Y
 9     I      [I, L, O, Y]         HIT  — I is in buffer (#1)
10     P      [P, L, O, Y]         miss — evict I (MRU), load P
11     P      [P, L, O, Y]         HIT  — P is in buffer (#2)
12     E      [E, L, O, Y]         miss — evict P (MRU), load E

Final pages: {E, L, O, Y}. Cache hits: 2.

Key insight: MRU tends to keep "old" pages and thrash the most recently used slot. This actually works well for sequential scans (repeated passes over a file) where LRU would fail.

Question 22

LRU Policy. Given 4 buffer frames and the access pattern:

A  B  T  P  H  M  O  A  A  B  A  C  B  E  A  F  G  B

What are the final pages in the buffer? How many cache hits?

Solution

LRU evicts the least recently used (accessed longest ago) page.

Step  Access  Buffer (LRU order, leftmost=LRU)  Action
 1     A      [A]                                 miss
 2     B      [A, B]                              miss
 3     T      [A, B, T]                           miss
 4     P      [A, B, T, P]                        miss (full)
 5     H      [B, T, P, H]                        miss — evict A (LRU)
 6     M      [T, P, H, M]                        miss — evict B (LRU)
 7     O      [P, H, M, O]                        miss — evict T (LRU)
 8     A      [H, M, O, A]                        miss — evict P (LRU)
 9     A      [H, M, O, A]                        HIT (#1)
10     B      [M, O, A, B]                        miss — evict H (LRU)
11     A      [M, O, B, A]                        HIT (#2) — A moves to MRU
12     C      [O, B, A, C]                        miss — evict M (LRU)
13     B      [O, A, C, B]                        HIT (#3) — B moves to MRU
14     E      [A, C, B, E]                        miss — evict O (LRU)
15     A      [C, B, E, A]                        HIT (#4) — A moves to MRU
16     F      [B, E, A, F]                        miss — evict C (LRU)
17     G      [E, A, F, G]                        miss — evict B (LRU)
18     B      [A, F, G, B]                        miss — evict E (LRU)

Final pages: {A, F, G, B}. Cache hits: 4.

Question 23

Clock Policy. 6 buffer frames. Pages loaded: A (pinned), B (pinned), C, D, E, F — all with ref bit = 1. Clock hand starts at frame 0 (A). Pages are unpinned immediately unless noted as "pinned."

A request for page G arrives (G must remain pinned). Which page is evicted? Then page F is requested. What happens?

Solution

Eviction search for G:

  1. Frame 0 (A): pinned — skip.
  2. Frame 1 (B): pinned — skip.
  3. Frame 2 (C): ref = 1, unpinned — clear ref to 0, advance.
  4. Frame 3 (D): ref = 1, unpinned — clear ref to 0, advance.
  5. Frame 4 (E): ref = 1, unpinned — clear ref to 0, advance.
  6. Frame 5 (F): ref = 1, unpinned — clear ref to 0, advance.
  7. Frame 0 (A): still pinned — skip.
  8. Frame 1 (B): still pinned — skip.
  9. Frame 2 (C): ref = 0, unpinned — EVICT C. Load G into frame 2.

Then F is requested: F is already in the buffer (frame 5) — cache hit. Set ref bit = 1.

Clock algorithm: Sweep the clock hand. Pinned pages are always skipped. For unpinned pages: if ref = 1, clear it and move on ("second chance"). If ref = 0, evict it. This approximates LRU.

Question 24

A page in the buffer pool has pin count = 3 and dirty bit = true. Can it be evicted? What happens when it is eventually evicted?

Solution

No, it cannot be evicted. A pin count > 0 means at least one query (in this case, 3 queries) is still actively using the page. The buffer manager must wait until all three queries call unpin() and the pin count reaches 0.

When eventually evicted: Since the dirty bit is true (the page was modified in memory), the page must be written back to disk before the buffer frame can be reused. This is an extra I/O cost. If the dirty bit were false, the frame could simply be overwritten since the disk already has the current version.

Pin count semantics: pin() increments the count (requestor starts using the page). unpin() decrements it (requestor is done). A page is eligible for eviction only when pin count = 0.

Question 25

True or False:

  1. The buffer manager sets the dirty bit.
  2. Sequential flooding happens with MRU.
  3. Clock policy with max ref counter of 1 approximates LRU.
  4. If a file is smaller than the buffer pool, LRU and MRU give the same hit rate.
Solution
  1. FALSE. The requestor (the query/operator using the page) sets the dirty bit when it unpins a modified page, not the buffer manager itself. The buffer manager only checks the dirty bit during eviction.
  2. FALSE. Sequential flooding is an LRU problem. When sequentially scanning a file larger than the buffer, LRU evicts the page that was accessed longest ago — which is exactly the next page you'll need on the next scan pass. MRU handles sequential scans well by keeping early pages cached.
  3. TRUE. The Clock algorithm with a single ref bit gives each page a "second chance" before eviction, which approximates LRU behavior. Pages that haven't been accessed recently will have their ref bit cleared and be evicted first.
  4. TRUE. When the entire file fits in the buffer pool, after the initial cold-start misses, all pages remain cached regardless of the eviction policy. No evictions ever occur, so LRU, MRU, Clock, and random all yield the same hit rate.
