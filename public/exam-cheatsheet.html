Side 1 (Front)

SQL Evaluation Order

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
  • WHERE: filters rows before grouping, NO aggregates allowed
  • HAVING: filters groups after aggregation, CAN use aggregates
  • SELECT aliases usable in ORDER BY, NOT in WHERE

SQL Aggregates

  • COUNT(*) includes NULLs, COUNT(col) excludes NULLs
  • SUM, AVG, MIN, MAX all ignore NULLs
  • Every non-agg column in SELECT must be in GROUP BY

Joins

TypeResult

LEFT OUTERAll left + NULL for non-matches
RIGHT OUTERAll right + NULL for non-matches

SQL Pitfalls

  • Integer division: 5/2=2. Fix: CAST(x AS FLOAT)
  • NULL=NULL is UNKNOWN, use IS NULL
  • NOT IN with NULLs → returns nothing! Use NOT EXISTS
  • DISTINCT + ORDER BY: ORDER BY cols must be in SELECT

Set Operations

UNION (dedup), UNION ALL (keep all), INTERSECT, EXCEPT

Subqueries

Keywords: IN, EXISTS, NOT EXISTS, ALL, ANY

  • Correlated: refs outer query, re-runs per row (slow)
  • Uncorrelated: runs once independently

B+ Tree Insert (MOST TESTED!)

  1. Search → correct leaf
  2. Insert key
  3. If overflow (> 2d keys):
    LEAF split → COPY UP middle key (stays in leaf!)
    INTERIOR split → PUSH UP middle key (removed from node!)
  4. May cascade to root

B+ Tree Delete

  1. Remove from leaf
  2. If underfull (< d): redistribute from sibling
  3. Can't redistribute → merge + pull down parent key

Composite Index Prefix Rule

Index (A,B,C) → can use for A, (A,B), (A,B,C)
CANNOT use for just B or C alone

Bulk Loading

Sort → fill leaves to fill factor → build up

Entries/leaf = floor(2d × fill_factor)
Leaf pages = ceil(records / entries_per_leaf)

KD Trees

  • Binary tree, alternating split dimensions
  • Level 0: X, Level 1: Y, Level 2: X...
  • Non-overlapping regions
  • Nearest neighbor: traverse down, backtrack checking other subtree if distance to split plane < best distance

R-Trees

  • Like B+ for spatial data
  • MBRs (Minimum Bounding Rectangles) CAN overlap
  • Insert: choose subtree needing LEAST expansion
  • Search may follow multiple paths
  • Always balanced

Buffer Pool

Frame metadata: Page ID, Dirty Bit, Pin Count

  • Pin > 0 → CANNOT evict
  • Dirty bit set by REQUESTOR, not buffer manager!
  • On eviction: if dirty, write to disk first

Clock Algorithm

  1. Check frame at hand
  2. Pinned? → skip, advance
  3. ref > 0? → decrement, skip, advance
  4. ref = 0? → EVICT! Load new page, set ref=1
  5. Hand stays where it stopped

Sequential Flooding

LRU + scan > buffer → 0 hits. MRU avoids this.
