1. SQL Deep Dive

1.1 Conceptual Evaluation Order

This is the single most important thing to understand about SQL. Every query is logically evaluated in this order:

FROM (identify tables + joins) → WHERE (filter individual rows) → GROUP BY (form groups) → HAVING (filter groups) → SELECT (compute output columns) → ORDER BY (sort) → LIMIT (truncate)

Why this matters for exams:

  • You cannot use an alias defined in SELECT inside WHERE (WHERE runs before SELECT)
  • WHERE filters rows before groups are formed. HAVING filters after aggregation
  • WHERE cannot contain aggregate functions. HAVING can
  • ORDER BY is the only clause that can reference SELECT aliases (it runs after SELECT)

1.2 Aggregation Rules

Aggregates collapse multiple rows into a single value:

  • COUNT(*) — counts all rows, including NULLs
  • COUNT(col) — counts non-NULL values in that column
  • COUNT(DISTINCT col) — counts distinct non-NULL values
  • SUM(col), AVG(col), MIN(col), MAX(col) — all ignore NULLs

The GROUP BY Rule (very commonly tested)

If you use an aggregate in SELECT, then every non-aggregated column in SELECT must appear in GROUP BY.

-- INVALID: name is not aggregated and not in GROUP BY
SELECT name, MAX(gpa) FROM Students;

-- VALID: dept is in GROUP BY
SELECT dept, MAX(gpa) FROM Students GROUP BY dept;

-- VALID: no aggregates at all, no GROUP BY needed
SELECT name, gpa FROM Students;

1.3 WHERE vs HAVING

Example: "Find departments where the average GPA of students with GPA above 2.0 is greater than 3.5"

SELECT dept, AVG(gpa)
FROM Students
WHERE gpa > 2.0          -- filters ROWS: only students with gpa > 2.0
GROUP BY dept             -- groups remaining rows by dept
HAVING AVG(gpa) > 3.5    -- filters GROUPS: only depts with high avg

Step by step:

  1. FROM Students — start with all student rows
  2. WHERE gpa > 2.0 — remove students with gpa ≤ 2.0
  3. GROUP BY dept — form groups by department
  4. HAVING AVG(gpa) > 3.5 — remove groups where average ≤ 3.5
  5. SELECT dept, AVG(gpa) — compute output

1.4 Joins Explained

Cross Join (Cartesian Product)

Every row of A paired with every row of B. If A has 5 rows and B has 3 rows, result has 15 rows.

SELECT * FROM A, B               -- implicit cross join
SELECT * FROM A CROSS JOIN B      -- explicit

Inner Join

Cross join + filter. Only rows where the ON condition is true.

SELECT * FROM A INNER JOIN B ON A.id = B.a_id

Natural Join

Auto-matches on ALL columns with the same name. Dangerous if tables share column names you didn't intend to join on.

SELECT * FROM A NATURAL JOIN B  -- matches all same-name columns

LEFT OUTER JOIN

Keeps ALL rows from the left table. If a left row has no match in the right table, right columns become NULL.

SELECT s.name, e.grade
FROM Students s LEFT JOIN Enrollments e ON s.sid = e.sid
-- Students with no enrollments appear with grade = NULL

Finding non-matches: WHERE e.sid IS NULL finds students with no enrollments.

Self Join

A table joined with itself. Must use aliases to distinguish the two "copies".

-- Find pairs of students in the same department
SELECT a.name, b.name
FROM Students a, Students b
WHERE a.dept = b.dept AND a.sid < b.sid

1.5 Set Operations

Both queries must have the same number of columns with compatible types.

1.6 Subqueries

IN / NOT IN

-- Sailors who reserved boat 103
SELECT sname FROM Sailors
WHERE sid IN (SELECT sid FROM Reserves WHERE bid = 103)

NOT IN with NULLs — Major Trap!

If the subquery returns ANY NULL value, NOT IN evaluates to UNKNOWN for every outer row, returning no results.

-- If subquery returns {1, 2, NULL}:
-- x NOT IN {1, 2, NULL} is UNKNOWN for ALL x
-- Use NOT EXISTS instead to be safe

EXISTS / NOT EXISTS

-- Courses with no failing students
SELECT c.name FROM Courses c
WHERE NOT EXISTS (
  SELECT 1 FROM Enrollments e
  WHERE e.cid = c.cid AND e.grade = 'F'
)

EXISTS returns TRUE if the subquery returns at least 1 row. NOT EXISTS returns TRUE if subquery returns 0 rows.

ALL / ANY

-- Students with GPA higher than ALL CS students
SELECT name FROM Students
WHERE gpa > ALL (SELECT gpa FROM Students WHERE dept = 'CS')

-- Students with GPA higher than ANY CS student (at least one)
SELECT name FROM Students
WHERE gpa > ANY (SELECT gpa FROM Students WHERE dept = 'CS')

Correlated vs Uncorrelated

Uncorrelated: The inner query is completely independent. It runs once, produces a result, then the outer query uses it.

Correlated: The inner query references a column from the outer query. It must re-execute for every row of the outer query. Think of it like a nested for-loop.

-- Correlated: inner query uses outer's s.dept
SELECT s.name FROM Students s
WHERE s.gpa > (SELECT AVG(gpa) FROM Students WHERE dept = s.dept)

1.7 CTEs and Views

-- CTE: temporary named result
WITH HighGPA AS (
  SELECT sid, name, gpa FROM Students WHERE gpa > 3.5
)
SELECT * FROM HighGPA WHERE name LIKE 'A%';

-- View: stored query (virtual table)
CREATE VIEW HighGPA AS SELECT sid, name, gpa FROM Students WHERE gpa > 3.5;
-- Can then query: SELECT * FROM HighGPA

Views don't store data — the underlying query runs each time the view is accessed.

1.8 SQL Pitfalls (Commonly Tested)

  • Integer division: 5/2 = 2 in SQL, not 2.5. Fix: CAST(5 AS FLOAT)/2 or 5 * 1.0 / 2
  • NULL comparisons: NULL = NULL is UNKNOWN, NULL != 5 is UNKNOWN. Always use IS NULL or IS NOT NULL
  • NULL in aggregates: COUNT(*) counts NULLs; COUNT(col), SUM, AVG ignore NULLs
  • DISTINCT + ORDER BY: When DISTINCT is used, ORDER BY can only reference columns in the SELECT list
  • HAVING without GROUP BY: Technically valid — treats entire table as one group
  • WHERE with aggregates: NEVER valid. WHERE COUNT(*) > 5 is a syntax error

2. Disks, Files & Records

2.1 Storage Hierarchy

From fastest/smallest to slowest/largest:

CPU Registers → L1/L2/L3 Cache → Main Memory (RAM) → SSD → HDD → Tape/Cloud

The DBMS manages data between RAM and disk. All data must be in RAM before the CPU can process it.

2.2 HDD vs SSD

HDD (Hard Disk Drive): Spinning magnetic platters with read/write heads on a moving arm. Data organized in concentric tracks, divided into sectors. Performance depends on:

  • Seek time: moving arm to correct track (slowest part, ~5-10ms)
  • Rotational latency: waiting for sector to spin under head (~2-4ms)
  • Transfer time: reading/writing data (fast once positioned)

SSD (Solid State Drive): Flash memory, no moving parts. No seek time or rotational latency. Much faster random reads. Write performance limited by erase-before-write requirement.

2.3 Pages

A page (or block) is the fundamental unit of data transfer between disk and memory. Typically 4KB or 8KB.

You always read or write a full page — you cannot read half a page. Even if you need one record, you must read the entire page containing it.

2.4 File Types

Heap File

Unordered collection of pages. Records inserted wherever there's free space. No sorting.

  • Pros: Fast inserts (just append), simple
  • Cons: Equality search requires scanning (B/2 avg IOs)

Implementation options:

  • Linked list: Header page with two linked lists — "full pages" and "free pages". To insert, scan free pages list for one with enough space.
  • Page directory: Header pages contain entries: (page_id, free_space_bytes). Can directly find a page with enough space. More efficient for inserts.

Sorted File

Pages maintained in sorted order by some key. Enables binary search for equality queries.

  • Pros: Fast equality search (log2(B) IOs), efficient range scans
  • Cons: Inserts/deletes expensive (must shift records to maintain order)

2.5 Page Layout: Slotted Pages

Inside each page:

  • Page header: number of records, free space pointer, slot directory
  • Records: grow from the TOP of the page downward
  • Slot directory: grows from the BOTTOM of the page upward

Each slot directory entry = (offset, length) pointing to a record in the page.

Why slotted pages? They support variable-length records, allow records to be moved within the page (just update the slot pointer) without changing external references, and enable compaction to reclaim space.

Record ID = (page_id, slot_number). This is stable even if the record moves within the page.

2.6 Record Formats

Fixed-Length Records

All fields have fixed size. Field i is at offset = sum of sizes of fields 0 through i-1.

Record: [field0 | field1 | field2 | field3]
         4 bytes  4 bytes  20 bytes 4 bytes
Field 2 offset = 4 + 4 = 8 bytes from start

Variable-Length Records

Some fields (VARCHAR, TEXT) have variable size. Need a header with field offset pointers.

Header: [ptr0 | ptr1 | ptr2 | ptr3 | field0 | field1 | field2 | field3]
Each pointer tells you where each field's data starts.

Null Bitmap

Instead of storing a special NULL value for each field, use a compact bitmap: one bit per field. If bit i is set, field i is NULL (its data is not stored, saving space).

2.7 Record Size Calculations (Exam Favorite!)

Example Calculation

CREATE TABLE Products(
  id INTEGER,          -- 4 bytes, always
  name VARCHAR(10),    -- 0 to 10 bytes
  price INTEGER,       -- 4 bytes, always
  desc CHAR(20)        -- 20 bytes, always (CHAR is fixed!)
);
Header = 8 bytes

Min record size = 8 + 4 + 0 + 4 + 20 = 36 bytes
Max record size = 8 + 4 + 10 + 4 + 20 = 46 bytes

Key distinction: CHAR(n) ALWAYS takes n bytes. VARCHAR(n) takes 0 to n bytes.

Records Per Page Calculation

For fixed-length records with bitmap:

Find largest n such that: n * record_size + ceil(n/8) ≤ page_size - page_header

The ceil(n/8) accounts for the bitmap (1 bit per slot, rounded to bytes).

3. IO Cost Model

3.1 Assumptions

  • B = number of data pages in the file
  • R = number of records per page
  • Only count disk I/Os (page reads + page writes)
  • Ignore CPU cost, sequential vs random distinction, prefetching
  • Every read or write of a page = 1 IO
  • Data must be in memory before operating on it

3.2 Complete IO Cost Table

OperationHeap FileSorted File
Scan all recordsBB
Equality searchB/2 average, B worstlog2(B)
Range searchB (full scan)log2(B) + matching pages
Insert2 (read last + write)log2(B) + B + B (find + shift reads + shift writes)
Delete(search cost) + 1 writelog2(B) + B + B

3.3 Understanding Each Cost

Heap File Insert = 2 IOs

Read the last page (1 IO) + write it back with the new record (1 IO). We always append to the end of a heap file, so we know exactly which page to read.

Heap File Equality Search = B/2 average

Linear scan through pages. On average, the record is halfway through the file. Worst case: B IOs (record on last page or doesn't exist).

Sorted File Equality = log2(B)

Binary search: read middle page, decide which half contains the key, repeat. Each step halves the search space and costs 1 IO.

Sorted File Insert = Very Expensive

Find the right position (log2(B) IOs), then shift all subsequent records to make room. In the worst case, this means reading and rewriting nearly all B pages.

3.4 Why Indexes?

Neither heap files (slow search) nor sorted files (slow insert/delete) are great for everything. Indexes aim to give fast search (like sorted) with reasonable insert/delete costs.

4. B+ Trees

4.1 Structure

A B+ tree of order d is a balanced search tree where:

  • Each interior node has d to 2d keys and d+1 to 2d+1 child pointers
  • Each leaf node has d to 2d entries (key + data pointer)
  • The root can have 1 to 2d keys (special exception)
  • All leaves are at the same depth (perfectly balanced)
  • Leaves are connected in a doubly-linked list for range scans

4.2 Key Formulas

Max keys per node = 2d

Min keys per node = d (root: 1)

Max fanout (children of interior node) = 2d + 1

Min fanout = d + 1 (root: 2)

Max entries in height-h tree = (2d+1)h-1 × 2d

Min entries in height-h tree = 2 × (d+1)h-2 × d (h ≥ 2)

4.3 Search

Starting at root, at each interior node:

  1. Find the correct child pointer by comparing your search key with the node's keys
  2. If search_key < first_key, follow leftmost pointer
  3. If search_key ≥ ki and search_key < ki+1, follow pointer between them
  4. If search_key ≥ last_key, follow rightmost pointer
  5. At the leaf: scan entries for the key

Cost: h IOs (h = height of tree), typically 2-3 for real databases.

4.4 Insert (With Splitting)

THE MOST IMPORTANT DISTINCTION ON THE EXAM:

Leaf split = COPY UP the middle key (key remains in the leaf)

Interior split = PUSH UP the middle key (key is removed from the node)

Why? Leaf nodes contain data pointers — the key MUST stay there to find the record. Interior nodes only route searches — the key can move up without losing any data.

Insert Example: Order d=1 (max 2 keys per node)

Starting with an empty tree, insert keys: 5, 10, 15, 20, 25

Insert 5: Create root leaf [5]

Insert 10: Root leaf becomes [5, 10]

Insert 15: Leaf [5, 10, 15] overflows! Split:

  • Left leaf: [5], Right leaf: [10, 15]
  • COPY UP 10 to new root: Root = [10]
  • Tree: [10] with children [5] and [10, 15]

Insert 20: Goes to right leaf [10, 15] → [10, 15, 20] overflows! Split:

  • Left: [10], Right: [15, 20]
  • COPY UP 15 to parent root: Root becomes [10, 15]
  • Tree: [10, 15] with children [5], [10], [15, 20]

Insert 25: Goes to rightmost leaf [15, 20] → [15, 20, 25] overflows! Split:

  • Left: [15], Right: [20, 25]
  • COPY UP 20 to parent: Root becomes [10, 15, 20] — overflows! Root split:
  • Interior split: PUSH UP 15. Left interior: [10], Right interior: [20]
  • New root: [15]

4.5 Delete (With Redistribution/Merge)

  1. Find the key in a leaf and remove it
  2. If leaf has ≥ d entries, done!
  3. If leaf becomes underfull (< d entries):
    • First try: Redistribute — borrow an entry from an adjacent sibling (if sibling has > d entries). Update parent key.
    • If can't redistribute: Merge with sibling — combine two underfull nodes into one, pull down the separating key from parent
  4. Merging may cause the parent to become underfull → cascade up

4.6 Bulk Loading

Why? Inserting N records one at a time = N × (search + possible splits) = very expensive.

Bulk loading algorithm:

  1. Sort all records by the search key
  2. Fill leaf pages to the desired fill factor (e.g., 2/3 full, 3/4 full)
  3. Build parent nodes bottom-up as leaves are filled

Fill factor calculation:

Entries per leaf = floor(2d × fill_factor)

Number of leaves = ceil(total_records / entries_per_leaf)

Bulk Loading Example

1000 records, order d=50 (max 100 entries per leaf), fill factor = 3/4

Entries per leaf = floor(100 × 0.75) = 75

Number of leaves = ceil(1000 / 75) = ceil(13.33) = 14 leaf pages

4.8 Clustered vs Unclustered

Clustered index: The order of entries in the leaf pages matches the physical order of records on disk. This means the data file is sorted by the index key.

  • Range scans are very efficient — consecutive records are on consecutive (or nearby) pages
  • After finding the start of a range, you can sequentially scan data pages

Unclustered index: The order of entries in the leaf pages does NOT match physical record order. Record pointers may point to scattered pages across the disk.

  • Range scans can be very expensive — each record might be on a different page
  • In the worst case, a range query touching k records needs k IOs (one per record)

Only ONE clustered index per table. The data can only be physically sorted in one order. All additional indexes must be unclustered.

4.9 B+ Tree IO Costs (Alt 2)

Equality Search (both clustered and unclustered):

  height IOs (traverse tree) + 1 IO (read data page) = height + 1 IOs

Range Search — Clustered:

  height + num_matching_leaf_pages + num_matching_data_pages

  Data pages are sequential, so relatively few IOs

Range Search — Unclustered:

  height + num_matching_leaf_pages + up to (num_matching_records) IOs

  Each record might be on a different data page!

Insert: height + 1 (read data page) + 1 (write data page) + 1 (write leaf) + possible splits

4.10 Composite Indexes

The Prefix Rule

An index on columns (A, B, C) sorts data by A first, then by B within each A value, then by C within each (A, B) pair.

The index can only be used efficiently if the query constrains a prefix of the index columns:

Query FilterCan Use Index?Why?
WHERE A = 5YesPrefix match
WHERE A = 5 AND B = 10YesPrefix match
WHERE A = 5 AND B = 10 AND C = 20YesFull match
WHERE B = 10NoB is not the first column
WHERE C = 20NoPrefix not constrained
WHERE B = 10 AND C = 20NoA (prefix) not constrained
WHERE A = 5 AND C = 20Partially (A only)B gap breaks the prefix

5. Spatial & Vector Indexes

5.1 Why Needed

B+ trees work for 1D ordered data. But many queries involve multiple dimensions:

  • "Find all restaurants within 1 mile of my location" (2D spatial)
  • "Find the 10 most similar documents to my query" (high-dimensional vectors)
  • SQL: WHERE x BETWEEN 1 AND 5 AND y BETWEEN 3 AND 8

A composite B+ tree index on (x, y) can filter on x but then must scan for y — not ideal.

5.2 KD Trees

A binary search tree where each level splits on a different dimension, alternating.

  • Level 0 (root): split on X
  • Level 1: split on Y
  • Level 2: split on X again
  • For K dimensions: cycle through dimensions 0, 1, ..., K-1, 0, 1, ...

Each node stores a data point. The point's coordinate in the current dimension determines the split.

Space is partitioned into non-overlapping regions — every point falls in exactly one region.

KD Tree Operations

Search: At each node, compare the query point's value for the current dimension with the node's value. Go left if less, right if greater or equal.

Insert: Same as search — traverse to find the correct position, then add the new node.

Range Query: At each node, check if the query range intersects the node's region. If yes, recurse into both children. If no, prune that subtree.

Nearest Neighbor Search

Finding the closest point to a query point Q:

  1. Traverse down the tree to find the leaf region containing Q
  2. Set the current best = the point found
  3. Backtrack up the tree: at each ancestor, check if the other subtree could contain a closer point
  4. Specifically: if the distance from Q to the splitting plane is less than the current best distance, search the other subtree
  5. Update best distance if a closer point is found

Application in AI: RAG (Retrieval-Augmented Generation) — documents are converted to embedding vectors, and nearest neighbor search finds the most relevant documents for a query. The LLM then generates answers using those documents as context.

5.3 R-Trees

An R-tree is like a B+ tree for multi-dimensional data.

  • Interior nodes: contain MBRs (Minimum Bounding Rectangles) — each MBR encloses all data in its subtree
  • Leaf nodes: contain actual spatial objects with their bounding boxes
  • Balanced (all leaves at same depth), like B+ trees
  • Has order d with fill constraints, like B+ trees

Key Difference from KD Trees

R-tree MBRs can overlap! This means a search query might need to follow multiple paths down the tree, not just one. KD tree regions are non-overlapping — each point falls in exactly one region.

R-Tree Operations

Search: Start at root. At each node, find all MBRs that overlap with the query region. Recurse into ALL matching children. (May follow multiple paths.)

Insert: Choose the subtree whose MBR needs the least expansion (smallest increase in area) to include the new object. If ties, choose the one with smallest area.

6. Buffer Management

6.1 Why Buffer Management?

The CPU can only work on data in RAM. Disk is slow. The buffer pool caches frequently-used disk pages in RAM to avoid repeated disk reads. The buffer manager decides which pages to keep and which to evict.

6.2 Buffer Pool Structure

The buffer pool is a large array of frames allocated at DBMS startup. Each frame holds exactly one page.

Each frame maintains three pieces of metadata:

  • Page ID: which disk page is currently in this frame
  • Dirty bit: has the page been modified since being read from disk?
  • Pin count: how many transactions/queries are currently using this page

6.3 Pin and Unpin Operations

Pin (Request a Page)

  1. Search buffer pool for the requested page
  2. If found (hit): increment pin count, return pointer to the frame
  3. If not found (miss):
    1. Find a frame to use: either an empty frame or evict a page with pin count = 0
    2. If the evicted page has dirty bit = 1, write it to disk first
    3. Read the requested page from disk into the frame
    4. Set pin count = 1, dirty bit = 0
    5. Return pointer to the frame

Unpin (Release a Page)

  1. Decrement pin count
  2. If the page was modified, the requestor tells the buffer manager to set dirty = 1

Critical Rules

  • A page with pin count > 0 CANNOT be evicted — someone is still using it
  • The requestor (not the buffer manager) sets the dirty bit
  • If a dirty page is evicted, it must be written to disk first
  • Pin count = number of current users, not total accesses

6.4 Replacement Policies

LRU (Least Recently Used)

When eviction is needed, evict the page that was accessed least recently (oldest last-access timestamp).

Intuition: "If you haven't been used recently, you're probably not needed soon."

Implementation: maintain a timestamp of last access for each frame. Evict the frame with the smallest timestamp (among those with pin count = 0).

MRU (Most Recently Used)

When eviction is needed, evict the page that was accessed most recently (newest last-access timestamp).

Intuition: "I just finished with this page, I don't need it anymore." Good for sequential access patterns where you process each page once and move on.

Sequential Flooding (Why MRU beats LRU for scans)

Scenario: Buffer has N frames. We're sequentially scanning a file with B > N pages.

With LRU: Each new page evicts the oldest page. The buffer cycles through pages from the scan, and pages from other queries get evicted. Result: 0 hits, every page is a miss.

With MRU: Each new page evicts the most recently used page (the one we just finished scanning). Earlier pages stay cached. If another query needs those earlier pages, they're still in the buffer. Result: first N-1 pages stay cached.

Bottom line: LRU is a good general-purpose policy but suffers from sequential flooding. MRU is better for sequential scans specifically.

Clock Policy

An efficient approximation of LRU. PostgreSQL uses Clock with max counter = 5.

Structure: Frames arranged in a circle. A "clock hand" sweeps around. Each frame has a reference counter.

On page access: Set the reference counter (or increment, up to max).

On eviction (need a frame):

  1. Look at the frame under the clock hand
  2. If pin count > 0: skip (can't evict), advance hand
  3. If ref counter > 0: decrement counter, advance hand (give a "second chance")
  4. If ref counter = 0: EVICT this page. Load the new page here.

The clock hand always starts from where it stopped last time (it doesn't reset).

6.5 Clock Policy Trace — Detailed Example

Setup: 4 frames, clock hand at position 0. All frames initially empty.

Access sequence: A, B, C, D, E, C, A, F

A: Empty frame 0. Load A, ref=1. Hand at 0.

B: Empty frame 1. Load B, ref=1. Hand at 0.

C: Empty frame 2. Load C, ref=1. Hand at 0.

D: Empty frame 3. Load D, ref=1. Hand at 0.

E: Buffer full! Need to evict. Hand at 0.

  Frame 0 (A, ref=1): ref-- → ref=0, advance to 1

  Frame 1 (B, ref=1): ref-- → ref=0, advance to 2

  Frame 2 (C, ref=1): ref-- → ref=0, advance to 3

  Frame 3 (D, ref=1): ref-- → ref=0, advance to 0

  Frame 0 (A, ref=0): EVICT A, load E. ref=1. Hand at 1.

Buffer: [E(1), B(0), C(0), D(0)]

C: Found in frame 2! Hit. ref = 1. Hand stays at 1.

Buffer: [E(1), B(0), C(1), D(0)]

A: Not in buffer. Need to evict. Hand at 1.

  Frame 1 (B, ref=0): EVICT B, load A. ref=1. Hand at 2.

Buffer: [E(1), A(1), C(1), D(0)]

F: Not in buffer. Need to evict. Hand at 2.

  Frame 2 (C, ref=1): ref-- → ref=0, advance to 3

  Frame 3 (D, ref=0): EVICT D, load F. ref=1. Hand at 0.

Buffer: [E(1), A(1), C(0), F(1)]

7. Exam Question Patterns

7.1 SQL Questions

Format: "Which of these queries correctly answers this question?" with 4 options.

Common traps:

  • Aggregate in SELECT but non-aggregated column missing from GROUP BY
  • WHERE used where HAVING is needed (or vice versa)
  • Integer division (missing CAST to FLOAT)
  • LEFT JOIN vs INNER JOIN — does the question need all rows from one table?
  • NOT IN vs NOT EXISTS with NULLs
  • Missing DISTINCT when duplicates should be removed

7.2 Disks/Files Questions

Common tasks:

  • Calculate min/max record sizes (watch CHAR vs VARCHAR!)
  • Calculate records per page (account for page header, bitmap)
  • Determine page directory vs linked list advantages
  • Count IOs for various operations on heap vs sorted files

7.3 B+ Tree Questions

Common tasks:

  • Trace an insertion with splits (remember: leaf COPY UP, interior PUSH UP)
  • "Is this a valid B+ tree?" — check order constraints, balanced depth, key ordering
  • Bulk loading calculations with fill factor
  • IO cost for equality/range queries on clustered vs unclustered
  • Composite index prefix rule

7.4 Buffer Management Questions

Common tasks:

  • Full LRU/MRU/Clock trace on a page access sequence
  • Count hits and misses
