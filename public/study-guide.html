1. SQL Fundamentals

Key Concept: Query Evaluation Order SQL has a specific conceptual evaluation order — it's NOT the order you write clauses:

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

SELECT / FROM / WHERE

The basic SQL query pattern retrieves data from tables, filters rows, and selects columns:

SELECT name, gpa
FROM Students
WHERE gpa > 3.5
ORDER BY gpa DESC
LIMIT 10;
  • DISTINCT — removes duplicate rows from output
  • ORDER BY — sorts output (ASC default, DESC for descending)
  • LIMIT — restricts number of output rows
  • Deterministic output requires ORDER BY + LIMIT together

Aggregates & Grouping

Aggregate functions compute over sets of values:

  • COUNT(*), COUNT(col), SUM(col), AVG(col), MIN(col), MAX(col)
  • Aggregates produce a single row of output (unless grouped)
SELECT dept, AVG(gpa) AS avg_gpa, COUNT(*) AS num_students
FROM Students
GROUP BY dept
HAVING COUNT(*) > 5;
Common Mistake: GROUP BY Rule Every non-aggregated column in SELECT must appear in GROUP BY.
SELECT dept, name, AVG(gpa) is INVALID without GROUP BY dept, name.
WHERE vs HAVING WHERE filters rows before grouping. HAVING filters groups after aggregation.
You can't use aggregates in WHERE: WHERE AVG(gpa) > 3.5 is INVALID.

Worked Example: Evaluation Order

Find departments where the average GPA of students who got an 'A' in any course is above 3.5.

SELECT S.dept, AVG(S.gpa) AS avg_gpa
FROM Students S JOIN Enrollments E ON S.sid = E.sid
WHERE E.grade = 'A'
GROUP BY S.dept
HAVING AVG(S.gpa) > 3.5;

Step-by-step evaluation:

  1. FROM: Cross product of Students × Enrollments
  2. ON: Keep rows where S.sid = E.sid (inner join)
  3. WHERE: Keep only rows where grade = 'A'
  4. GROUP BY: Partition remaining rows by dept
  5. HAVING: Keep groups where AVG(gpa) > 3.5
  6. SELECT: Output dept and avg_gpa for surviving groups

Key Concept: Pages A page (or block) is the unit of disk I/O. Typically 4 KB. The DBMS reads/writes entire pages at a time — never individual records.

HDD Internals

Spinning disks have platters with concentric tracks divided into sectors. Access time = seek time (move arm) + rotational delay (wait for sector) + transfer time (read data). Sequential access is much faster than random access because seek + rotation are avoided.

File Organizations

TypeStructureBest For
Heap FileUnordered pagesInserts (fast), full scans
Sorted FilePages in sorted orderBinary search, range queries

Slotted Page Layout

Step-by-Step: How Slotted Pages Work
  1. Header at the very start: stores number of slots, free space pointer
  2. Slot directory at the bottom of the page, growing upward — each slot entry stores (offset, length) of a record
  3. Records at the top of the page, growing downward
  4. Free space is in the middle between records and slot directory
  5. To delete: remove the slot entry, compact records if desired
  6. Supports variable-length records because slot directory stores offsets

Record Layout

  • Fixed-length: field offset = sum of preceding field sizes. Simple arithmetic.
  • Variable-length: header stores offsets/pointers to each field's start location.
  • Null bitmap: 1 bit per field — 1 if NULL, 0 if present. Compact representation.
Worked Example: Record & Page Calculations

CREATE TABLE Products(id INTEGER, name VARCHAR(10), price FLOAT, desc CHAR(20))

Sizes: INTEGER = 4B, FLOAT = 4B, CHAR(20) = 20B, VARCHAR(10) = 0–10B, Header = 8B

Min record size = 8 + 4 + 0 + 4 + 20 = 36 bytes (VARCHAR empty)

Max record size = 8 + 4 + 10 + 4 + 20 = 46 bytes (VARCHAR full)

On a 1024-byte page with 12B page header and 4B per slot entry:

Max records/page (using min record size): ⌊(1024 − 12) / (36 + 4)⌋ = ⌊1012 / 40⌋ = 25

Min records/page (using max record size): ⌊1012 / (46 + 4)⌋ = ⌊1012 / 50⌋ = 20

3. IO Cost Analysis

Key Concept: Cost Model Assumptions
  • Only count disk I/Os (reads and writes)
  • Ignore CPU cost, ignore sequential vs. random distinction
  • Ignore prefetching
  • B = number of pages in the file
  • R = number of records per page

Cost Comparison Table

Range SearchBlog₂(B) + matching pages

Common Mistake: Heap Insert Cost Heap insert is 2 IOs, not 1. You must READ the last page into memory, then WRITE it back after adding the record. Always count both read and write.
Worked Example: IO Cost Comparison

A file has B = 500 pages, R = 20 records/page.

Heap is great for inserts, terrible for search. Sorted is great for search, terrible for inserts. This motivates indexes — the best of both worlds.

4. B+ Trees

Key Concept: B+ Tree Structure
  • Interior nodes: keys + pointers to children. No data.
  • Leaf nodes: keys + pointers to actual records. Connected via doubly linked list.
  • All leaves at same depth (perfectly balanced).
  • Order d: each node has [d, 2d] keys (root: [1, 2d]).
  • Fan-out: interior node has [d+1, 2d+1] children.

Search

Start at root. At each node, compare search key against node's keys. Follow the appropriate pointer (left of smaller keys, right of larger). Repeat until reaching a leaf. Check if key exists in the leaf.

Insert Algorithm

Step-by-Step: B+ Tree Insert
  1. Search for the correct leaf node
  2. If leaf has space → insert the key in sorted order. Done.
  3. If leaf is full (has 2d keys) → SPLIT:
    • Create new leaf, distribute keys: first d in old leaf, last d in new leaf
    • COPY the middle key up to parent (leaf keys are copied, not moved!)
  4. If parent is full → split parent too:
    • For interior nodes, PUSH the middle key up (it moves, not copied)
  5. May cascade splits all the way to root → tree grows one level taller
CRITICAL: Copy Up vs Push Up Leaf split: middle key is COPIED up (it stays in the leaf too, because leaves hold actual data pointers).
Interior split: middle key is PUSHED up (it moves out of the node entirely).
This is the #1 most common B+ tree mistake on exams!

Delete Algorithm (Brief)

  1. Find and remove the entry from the leaf
  2. If leaf is underfull (fewer than d keys) → try redistribute from sibling
  3. If redistribution fails → merge with sibling, pull down separator from parent
  4. Parent may become underfull → cascade up

Bulk Loading

  1. Sort all records by key
  2. Fill leaves left to right with a fill factor (e.g., 2/3 full to leave room for inserts)
  3. Build internal nodes bottom-up
  4. Much faster than inserting one at a time — avoids repeated root-to-leaf traversals

5. B+ Tree Advanced Topics

Clustered vs Unclustered

Key Concept: Clustering Clustered: leaf order ≈ physical data order on disk. Built on top of a sorted file. Range scans read sequential pages.
Unclustered: leaf order ≠ physical order. Each record pointer may go to a different page. Range scans cause random I/O.

Only ONE clustered index per table — data can only be physically sorted one way.

Composite (Compound) Indexes

The Prefix Rule Index on (A, B) sorts by A first, then B within the same A value.
  • Can use for: queries on A, or A AND B
  • Cannot use for: queries on B alone
Think of a phone book sorted by (last name, first name). You can look up by last name, or by both. You can't efficiently look up by first name alone.

IO Costs with B+ Trees

OperationClustered (Alt 2)Unclustered (Alt 2)
Equality Searchheight + 1 IOheight + 1 IO
Range Searchheight + leaf pages + data pagesheight + leaf pages + up to 1 IO per matching record
Worked Example: Clustered vs Unclustered Range Query

Table: 100,000 records, 100 records/page → 1,000 data pages. B+ tree height = 2. Query returns 500 records across 5 leaf pages.

Clustered: 500 records at 100/page = 5 data pages (sequential!)

Cost = 2 (tree traversal) + 5 (leaf pages) + 5 (data pages) = 12 IOs

Unclustered: 500 records could be on up to 500 different pages

Cost = 2 + 5 + 500 = 507 IOs (worst case)

Heap scan: 1,000 IOs. So unclustered (507) is still better than heap, but clustered (12) is dramatically better.

When Unclustered Is Worse Than Heap Scan If the range query returns a large fraction of the table, unclustered index can be worse than a sequential heap scan. PostgreSQL's query optimizer knows this and will choose a sequential scan when it estimates the selectivity is too high.

6. Spatial Indexes

Key Concept: Why Spatial Indexes? B+ trees handle 1D ordered data. For multi-dimensional queries (find all restaurants within 5 miles, find stars in a region of sky), we need spatial indexes that understand 2D+ geometry.

KD Trees (K-Dimensional Trees)

  • Binary search tree that alternates splitting dimension at each level
  • Level 0: split on X, Level 1: split on Y, Level 2: split on X, ...
  • Each node stores a data point and partitions space into two half-planes
  • Search: compare appropriate dimension at each level, go left or right
  • Insert: search for position, add as new leaf
Worked Example: KD Tree Construction

Insert points: A(7,2), B(5,4), C(9,6), D(2,3), E(4,7)

Insert A(7,2): Root. Splits on X at x=7.

Insert B(5,4): 5 < 7 → go left. B becomes left child. Splits on Y at y=4.

Insert C(9,6): 9 > 7 → go right. C becomes right child. Splits on Y at y=6.

Insert D(2,3): 2 < 7 → left to B. 3 < 4 → go left of B. D becomes left child of B. Splits on X at x=2.

Insert E(4,7): 4 < 7 → left to B. 7 > 4 → go right of B. E becomes right child of B. Splits on X at x=4.

Tree structure:

Space partitioning: A splits the plane vertically at x=7. Left of A, B splits horizontally at y=4. Below B, D is at (2,3). Above B, E is at (4,7). Right of A, C is at (9,6).

Nearest Neighbor Search

Step-by-Step: KD Tree Nearest Neighbor
  1. Traverse tree as if searching for the query point — reach a leaf (candidate)
  2. Record distance to this candidate as "best so far"
  3. Backtrack: at each parent, check if the other subtree could contain a closer point
  4. If the perpendicular distance to the splitting plane < best distance → explore other subtree
  5. Update best distance if a closer point is found

Application: RAG (Retrieval-Augmented Generation) in LLMs — documents converted to vectors, nearest neighbor finds relevant docs.

R-Trees (Rectangle Trees)

  • Like B+ trees but for multi-dimensional data — balanced (all leaves same depth)
  • Interior nodes contain MBRs (Minimum Bounding Rectangles) that enclose all children's objects
  • Leaf nodes contain actual spatial objects with their bounding boxes
  • MBRs can overlap — unlike KD tree partitions

7. Buffer Management

Key Concept: Buffer Pool The buffer pool is a large array of frames in RAM. Each frame can hold one page. It's the bridge between disk and CPU — everything must be in the buffer pool before the CPU can process it.

Frame Metadata

FieldMeaning
Page IDWhich disk page is currently in this frame
Dirty BitHas the page been modified since read from disk?
Pin CountNumber of queries currently using this page

Operations

  • Pin (request page): If in buffer → increment pin count. If not → read from disk into a free frame, set pin count = 1.
  • Unpin (release page): Decrement pin count. Optionally set dirty bit.
  • If buffer full → must evict a page with pin count = 0. If dirty → write to disk first.
  • Page with pin count > 0 CANNOT be evicted.

Replacement Policies

LRU (Least Recently Used)

Evict the page that was unpinned longest ago. Intuition: if you haven't used it recently, you probably won't need it soon.

LRU's Problem: Sequential Flooding When sequentially scanning a file larger than the buffer, LRU evicts the oldest page — but that's the next page you need for a repeated scan! Every access becomes a miss. LRU gives 0% hit rate for repeated sequential scans on large files.

MRU (Most Recently Used)

Evict the page that was unpinned most recently. Better for sequential scans — you just finished with that page, so evict it.

Clock Policy

Step-by-Step: Clock Replacement
  1. Frames arranged in a circle with a "clock hand" pointer
  2. Each frame has a reference counter (set to 1 when page is pinned/accessed)
  3. When eviction needed, the clock hand sweeps:
    • If pin count > 0 → skip (can't evict)
    • If ref counter > 0 → decrement counter, move on (second chance)
    • If ref counter = 0 → EVICT this page
  4. PostgreSQL uses Clock with max counter of 5

Clock approximates LRU — recently used pages have higher counters and survive longer.

Buffer: 3 frames. Access pattern: A B C D A B C D

8. Practice Problems

Problem 1: SQL

Given: Books(bid, title, author, year) and Checkouts(bid, uid, checkout_date)

(a) Which query correctly finds authors whose books have been checked out more than 10 times total?

Option A:

SELECT B.author
FROM Books B JOIN Checkouts C ON B.bid = C.bid
WHERE COUNT(*) > 10
GROUP BY B.author;

Option B:

SELECT B.author
FROM Books B JOIN Checkouts C ON B.bid = C.bid
GROUP BY B.author
HAVING COUNT(*) > 10;

Option C:

SELECT B.author, COUNT(*) AS cnt
FROM Books B, Checkouts C
WHERE B.bid = C.bid
GROUP BY B.author
HAVING cnt > 10;
Show Solution

Answer: Option B

  • Option A is INVALID: You cannot use COUNT(*) in WHERE. Aggregates only work in HAVING or SELECT.
  • Option B is CORRECT: Groups by author, then filters groups where count > 10 using HAVING.
  • Option C is WRONG (in standard SQL): You cannot reference a column alias (cnt) in the HAVING clause in standard SQL. Some databases (MySQL) allow this, but PostgreSQL does not. The correct HAVING would be HAVING COUNT(*) > 10.

(b) Write a query using a CTE to find the title of the most checked-out book.

Show Solution 
WITH BookCounts AS (
  SELECT B.bid, B.title, COUNT(*) AS cnt
  FROM Books B JOIN Checkouts C ON B.bid = C.bid
  GROUP BY B.bid, B.title
)
SELECT title
FROM BookCounts
ORDER BY cnt DESC
LIMIT 1;

INT = 4 bytes, FLOAT = 4 bytes, CHAR(n) = n bytes, VARCHAR(n) = 0 to n bytes. Record header = 8 bytes. Page size = 4096 bytes, page header = 24 bytes, slot entry = 6 bytes.

(a) What is the maximum record size?

(b) What is the minimum number of records that fit on one page?

(c) If we have 50,000 records and use the minimum packing, how many pages does the heap file need?

Show Solution

(a) Max record size = header + INT + CHAR(20) + FLOAT + VARCHAR(15 max)
= 8 + 4 + 20 + 4 + 15 = 51 bytes
